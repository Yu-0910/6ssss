# 全年度「規定用・全員用CSV分割」実行計画書

---

## 完全版概要（ゴール：サイト頁への反映）

**本計画のゴール**は、規定用・全員用CSVの分割とランキングJSONの再構成を**全年度**に適用し、その結果を**ランキングのサイト頁に反映すること**である。

### ゴールの定義

- **サイト頁**: ランキングページ（例: `/ranking/2025/pl`, `/ranking/1975/cl` 等）。
- **反映**: 表示時に読むJSONが、規定必須指標は規定到達者のみ・カウント系は全員のデータからビルドされたものであり、負荷と読み込み時間が抑えられた状態で正しく表示されること。
- **完了条件**: 全対象年度・リーグで上記が成り立ち、運用手順とドキュメントが整っていること。

### データからサイト頁までの流れ

```
[生CSV] master_csv__import_1950_2024 / master_csv
    ↓ compute_metrics_all_seasons.py
[計算済みCSV] master_csv_calculated（全員用）
    ↓ create_qualifying_csv_all_years.py（Phase 1）
[規定到達版CSV] master_csv_calculated/*_qualifying.csv
    ↓ build_rankings_from_calculated.py / build_rankings_2025_PL_full.py（Phase 2）
[ランキングJSON] public/data/rankings/{年}/{リーグ}/{指標}.json
    ↓ アプリが取得（/data/rankings/... 経由）
[サイト頁] ランキングページに表示 ← ここがゴール
```

### 計画の構成（Phase とサイト頁への接続）

| Phase | 内容 | サイト頁への接続 |
|-------|------|------------------|
| **1** | 規定打席到達版CSV生成を全年度対応にする。規定必須指標に必要な列のみを組み込んだ規定用CSVを、全年度・全リーグ（PRE春秋含む）で出力する。 | 規定用CSVが無いと、Phase 2 で規定あり版JSONを「規定到達者のみ」で作れない。Phase 1 が完了して初めて、サイト頁で規定必須指標が軽量なJSONで表示できる。 |
| **2** | ランキングビルドで規定用CSVを参照する。規定必須指標は `*_qualifying.csv` があればそれを入力にJSONを生成し、カウント系は従来どおり全員用CSVから生成する。 | ここで出力するJSONを、サイト頁のランキング表示が読みにいく。Phase 2 の完了により、サイト頁に「規定用・全員用を分けたデータ」が反映される。 |
| **3** | 入力・出力ディレクトリの統一と検証。パスの整理とサンプル年度での検証により、Phase 1–2 の再現性を確保する。 | 運用が安定し、サイト頁に渡すJSONの出所と品質が明確になる。 |
| **4** | サイト側の規定フィルタ整理と後方互換。規定適用済みJSONではフィルタが実質 no-op になることを前提に、古いJSONでも正しく表示されるようにする。 | サイト頁の表示ロジックと、ビルド済みJSONの関係を整理し、ゴール（正しく軽い表示）を維持する。 |
| **5** | ドキュメント・運用手順の整備。実行順序・分割実行例・計画書参照を明文化する。 | サイト頁へ反映した後も、データ更新や再ビルドを誰でも再現できるようにする。 |

### 指標の整理（サイト頁でどう使うか）

- **規定必須指標**（OPS, 打率, 出塁率, 長打率, IsoP, IsoD, BB%, K%, BB/K, RC, XR, BABIP, SecA, TA, NOI, GPA）  
  → 規定用CSVからビルドしたJSONをサイト頁が読む。表示時は規定到達者のみが並ぶ。
- **カウント系指標**（安打, 本塁打, 打点, 試合, 打席, 打数, 単打, 二塁打, 三塁打, 得点, 四球, 敬遠, 死球, 三振, 塁打, 盗塁, 盗塁死, 犠打, 犠飛, 併殺打）  
  → 全員用CSVからビルドしたJSONをサイト頁が読む。表示時は全選手が対象。

### 規定ルールの記録（サイト・ビルドで参照するもの）

- 規定の細則（1950年代PL中心）は **lib/ranking/qualifyingPA.ts**（サイト側）と **scripts/qualifying/qualifying_rules.py**（ビルド側）および **_data/npb_regulation_table.csv** にあり、Phase 1 の規定到達版CSV生成でも同じルールを使う。

### 実行順序とゴール到達

1. **Phase 1** を年度範囲を絞って実行 → 規定用CSVを用意する。
2. **Phase 2** を同じ範囲で実行 → ランキングJSONを再生成する。
3. アプリを起動し、該当するランキング頁（例: `/ranking/1975/pl`）を開く → **サイト頁に反映されていることを確認する**（ゴールの確認）。
4. 問題なければ範囲を広げ、Phase 3–5 で検証・運用・ドキュメントを整える。

**完了時点で**: 全対象年度のランキング頁が、規定用・全員用を分けたデータに基づき、負荷を抑えつつ正しく表示されている状態がゴールである。

## 目的

ランキング表示時の負荷と読み込み時間を減らすため、**規定打席到達者用CSV**と**全員用CSV**を分けて用意し、ビルド時に規定必須指標は規定用CSVから、カウント系指標は全員用CSVからJSONを生成する方式を**全年度**に適用する。

- **現状（2025年のみ）**: `create_qualifying_csv_2025.py` で規定到達版CSVを作成し、`build_rankings_2025_PL_full.py` が規定必須指標でそれを参照している。
- **目標**: 1936年〜2025年（および戦前PRE・春秋分割含む）の全シーズンで同方式を適用し、分割して実行可能な Phase で進める。

### CSVの列構成方針：必要な指標のみ組み込む

**規定用CSV**と**全員用CSV**は、それぞれで使う指標に必要な列だけを組み込む。

| CSV種別 | 対象指標 | 組み込む列の考え方 |
|---------|----------|---------------------|
| **規定打席到達版CSV** | 規定必須指標（OPS, 打率, 出塁率, 長打率, IsoP, IsoD, BB%, K%, BB/K, RC, XR, BABIP, SecA, TA, NOI, GPA） | 選手識別（year, league, team, player_id, player_name_ja, player_name_en）および、上記の率・指標を算出するのに**必要な列のみ**（例: G, PA, AB, R, H, 2B, 3B, HR, TB, RBI, BB, IBB, HBP, SO, GDP, AVG, OBP, SLG, OPS と計算済み列など）。カウント系のみで使う列（本塁打単体でランキングするとき以外に不要な列）は省いてよい。 |
| **全員用CSV** | 規定不要のカウント系指標（安打, 本塁打, 打点, 試合, 打席, 打数, 単打, 二塁打, 三塁打, 得点, 四球, 敬遠, 死球, 三振, 塁打, 盗塁, 盗塁死, 犠打, 犠飛, 併殺打） | 選手識別列および、上記カウント系指標のランキング・算出に**必要な列のみ**。規定必須指標用の計算済み列（OPS, 打率, 出塁率など）は含めなくてよい。 |

- こうすると各CSVのサイズと、ビルド時の読み込み量を抑えられる。
- 現状のように「全列コピー」でも動作はするが、計画上は**必要な指標（列）のみを組み込む**ことを方針とする。実装時はまず全列で出力し、のちに列を絞る段階を設けてもよい。

## 規定ルールの記録（1950年代PLを中心とした細則）

**「アプリが毎回取り出している」規定の正体**は、ランキング表示時に参照している**コードとデータ**です。会話で伝えていただいた細かいルールは、次の場所に**実装とコメント**として残っています。

### 記録の所在

| 場所 | 役割 | 内容 |
|------|------|------|
| **lib/ranking/qualifyingPA.ts** | アプリ側（ランキング表示時の規定判定・フィルタ） | 年度別試合数（TEAM_GAMES_DATA）、1950–1958年規定打数（AB_RULES_1950_1958）、1951年PLチーム別AB（PL_1951_TEAM_AB_THRESHOLD）、1952年PLチーム群・順位別AB（PL_1952_TEAM_AB_THRESHOLD）、1966/1967年PLチーム別PA、calculateMinPA / get1950_1958AB / get1950MinGames 等。**コメントで「当時のチーム名→現在のCSVチーム名」対応も記載。** |
| **scripts/qualifying/qualifying_rules.py** | ビルド側（ランキングJSON生成時の規定判定） | get_qualifying_rule(year, league, season_code)、calc_qual_threshold、is_player_qualified。1950–1958年CL/PLの basis（AB_FIXED, AB_TEAMGAMES_MULT, AB_TEAM_GROUP, PA_TEAMGAMES_MULT, PA_FIXED）と threshold / team_groups。PL_1952_TOP4_TEAMS（南海・毎日・西鉄・大映）。 |
| **_data/npb_regulation_table.csv** | 参照用サマリ | 年度・リーグごとの qualifying_pa / qualifying_ip の説明と備考（例: 1952 PL「パ・リーグは総当り終了後、上位4球団で…」）。細則の数値はコード側が正。 |

### 1950年代パ・リーグの細則（コードに書かれているルール）

- **1950年PL**: 規定打数（AB）固定 **300**。試合数条件なし。
- **1951年PL**: 規定打数は**チーム別**（チーム試合数×2.5 切り捨て）。南海104→260、西鉄105→262、毎日110→275、大映101→252、阪急96→240、東急102→255、近鉄98→245。**CSVのチーム名は現在名**（福岡ソフトバンクホークス＝南海、埼玉西武＝西鉄、千葉ロッテ＝毎日、オリックス＝阪急、日本ハム＝東急、近鉄バファローズ＝近鉄 等）でマッピング済み。
- **1952年PL**: 規定打数は**順位によるチーム群**。上位4球団（南海・毎日・西鉄・大映）**300**、下位3球団（阪急・東急・近鉄）**270**。チーム名は現在名でマッピング済み。
- **1953年PL**: 規定打数固定 **300**。
- **1954–1955年PL**: 規定打数固定 **360**。
- **1956年PL**: 規定打数固定 **400**。
- **1957年PL**: **規定打席（PA）**＝チーム試合数×3.1（全6チーム132試合→409打席、切り捨て）。
- **1958年PL**: 規定打席（PA）固定 **400**。

このほか、1950年CLは**試合数≥100**の追加条件（get1950MinGames）、1966/1967年PLは**チーム別規定打席**（チーム試合数×3.1 切り捨て）が qualifyingPA.ts に定義されています。

Phase 1 の規定到達版CSV生成では、**上記と同じルール**で行ごとにしきい値を決める必要があるため、ビルド側は `qualifying_rules.py` の `get_qualifying_rule` と `calc_qual_threshold` を利用し、チーム名は CSV の「チーム」列をそのまま渡す（必要なら qualifyingPA.ts と同様の「現在のチーム名」マッピングを Python 側にも持つ）形にしてください。

## Phase 1: 規定打席到達版CSV生成スクリプトの全年度対応

### 目的

`create_qualifying_csv_2025.py` を拡張し、**任意の年度・リーグ**（およびPRE春秋）に対し、規定打席到達者だけを抽出した `batting_YYYY_LEAGUE_qualifying.csv` を生成できるようにする。

### 作業内容

1. **新スクリプト作成**（例: `scripts/create_qualifying_csv_all_years.py`）
   - 入力: `_data/master_csv_calculated/` 内の `batting_YYYY_LEAGUE_from_master.csv`（または `batting_YYYY_PRE_spring/fall_from_master.csv`）を列挙。
   - 規定打席（minPA）の算出: 既存の `qualifying_rules.py`（`get_qualifying_rule`, `calc_qual_threshold`）および `games_map`（`build_games_per_team_map.py` 出力）を利用。年度・リーグ・チーム別の特別ルール（1952 PL, 1966/1967 PL 等）に対応。
   - 行のフィルタ: PA または AB（1950–1958 年等の規定打数年度）がしきい値以上なら規定到達として残す。チーム別しきい値の年度は行ごとにチームを参照して判定。
   - 出力: 入力と同じディレクトリに `batting_YYYY_LEAGUE_qualifying.csv` を保存（入力ファイル名の `from_master` の直前に `_qualifying` を挟む等、命名規則を統一）。**列は規定必須指標に必要な列のみに限定する**（上記「CSVの列構成方針」参照）。まずは全列で出力し、のちに列を絞る実装にしてもよい。
2. **コマンドライン引数**
   - `--input-dir`: 入力CSVのディレクトリ（デフォルト: `_data/master_csv_calculated`）。
   - `--year`: 対象年度（省略時は全年度）。
   - `--league`: 対象リーグ（省略時は全リーグ）。
   - `--dry-run`: 実際には書き込まず、対象ファイルと件数だけ表示。
3. **実行単位**
   - まずは `--year 2024` などで単年検証し、問題なければ `--year 1950 --league PL` のように年度・リーグを区切って段階実行。最後に引数なしで一括実行可能にする。

### 想定ネガティブポイントと対処法

| ネガティブポイント | 対処法 |
|--------------------|--------|
| **年度・リーグ数が多く、一括実行が長時間になる** | ・`--year` / `--league` で範囲を絞り、 decade 単位（例: 1950–1959）やリーグ単位で分割実行する。・ログに「何年度何リーグを何件→何件にした」を出し、進捗と結果を追えるようにする。 |
| **規定ルールが年度・リーグで異なる（PA/AB、チーム別しきい値等）** | ・規定しきい値の算出は既存の `qualifying_rules.py` と `games_map` に一本化する。・1952 PL・1966 PL・1967 PL 等は行の「チーム」列から `calc_qual_threshold(rule, team_games, team_name=...)` を呼び、行ごとにしきい値を決めて比較する。・PA列がない年度はAB列で判定するなど、列の有無をチェックしてフォールバックする。 |
| **PRE（戦前）・春秋分割シーズンのファイル名・league_key が特殊** | ・`scripts/lib/filename_parser.py` の `parse_batting_filename()` でパースし、`year`, `league_key`, `season_info` を取得。・`batting_1936_PRE_spring_from_master.csv` 等に対応する出力名を一意に決める（例: `batting_1936_PRE_spring_qualifying.csv`）。・`get_qualifying_rule(year, league, season_code)` の `season_code` に `1936s` / `1936f` を渡す。 |
| **対象CSVが存在しない年度・リーグがある** | ・入力ディレクトリを glob で走査し、存在する `batting_*_from_master.csv` のみ処理する。・該当する入力ファイルがない年度・リーグはスキップし、ログに「スキップ: 入力なし」と出す。・欠損年度の一覧を `--dry-run` で事前確認できるようにする。 |
| **列名のゆらぎ（PA / 打席 / pa、AB / 打数）** | ・行から数値を取る際、候補列名リスト `['PA','pa','打席']` / `['AB','ab','打数']` を順に試し、最初に取得できた値を使う。・ヘッダー行を1行読んで列名を確認し、存在しない列はスキップする。 |
| **規定ルールが未定義の年度（例: 1945年）** | ・`get_qualifying_rule()` が None を返す場合は「規定ルールなし」としてスキップし、規定到達版CSVは生成しない。・ログに「規定ルール未定義のためスキップ」と明記する。 |

## Phase 2: ランキングビルドでの規定用CSV参照（全年度）

### 目的

ランキングJSONを生成するビルド処理で、**規定打席必須の指標**については、規定到達版CSV（`*_qualifying.csv`）が存在すればそれを入力に使い、存在しなければ従来どおり全員用CSV＋minPAフィルタで生成するようにする。対象は **build_rankings_from_calculated.py**（主に2024年以前）および **build_rankings_all_from_yearly_dir.py** 等、全年度を扱うスクリプト。

### 作業内容

1. **build_rankings_from_calculated.py の変更**
   - 各 `batting_YYYY_LEAGUE_from_master.csv` を処理する前に、同じディレクトリに `batting_YYYY_LEAGUE_qualifying.csv`（および PRE_spring/fall の場合は対応する qualifying ファイル）があるか確認する。
   - 規定必須指標（`METRICS_REQUIRE_QUALIFYING_PA_BY_NAME` に含まれる指標）の「規定あり版」JSONを生成する際、qualifying CSV が存在すればそれを読み、`min_pa=0` でランキング生成。存在しなければ従来どおり全員用CSVを読み、既存の minPA でフィルタして生成。
   - カウント系指標および「規定なし版」（`*_all.json`）は従来どおり全員用CSVのみ使用する。
2. **build_rankings_2025_PL_full.py**
   - 現状のまま（既に規定用CSVを参照している）でよい。Phase 1 で 2025 の規定用CSVは既に存在する前提。
3. **build_rankings_all_from_yearly_dir.py を使っている場合**
   - 上記と同様に、規定必須指標について qualifying CSV の有無を確認し、あればそれを入力にする分岐を追加する。

### 想定ネガティブポイントと対処法

| ネガティブポイント | 対処法 |
|--------------------|--------|
| **規定用CSVがない年度は従来挙動に戻す必要がある** | ・qualifying CSV の存在チェックを「ファイルが存在する場合のみ」の分岐にし、存在しなければこれまで通り全員用CSV＋minPAで生成する。・後方互換性を保つため、既存の minPA 計算ロジックは削除せずそのまま残す。 |
| **入力ディレクトリが master_csv_calculated と master_csv で混在している** | ・規定用CSVの探索パスを「処理対象の全員用CSVと同じディレクトリ」に限定する。・build_rankings_from_calculated.py の入力ディレクトリが `_data/master_csv_calculated` なら、規定用CSVもそこから読む。・他スクリプトで別ディレクトリを参照している場合は、そのスクリプト内で「同じ input_dir 内の qualifying ファイル」を参照するようコメントで明記する。 |
| **2025年は build_rankings_from_calculated では処理しない（スキップしている）** | ・2025年は引き続き build_rankings_2025_PL_full.py のみで処理する前提でよい。・ドキュメントに「2024年以前: build_rankings_from_calculated / 2025年: build_rankings_2025_PL_full」と記載し、運用で混乱しないようにする。 |
| **PRE や春秋分割の league_key とファイル名の対応** | ・`parse_batting_filename()` で得た `league_key`（例: PRE_spring）と一致する qualifying ファイル名を組み立てる（例: `batting_1936_PRE_spring_qualifying.csv`）。・既存の `build_rankings_from_calculated` が参照する CSV 名規則と揃える。 |
| **規定用CSVの行数が0になる年度がある** | ・規定到達者が誰もいない場合、規定用CSVは空またはヘッダーのみになる可能性がある。・ビルド側で「規定用CSVを読んだが行数が0」の場合は、規定あり版JSONは空配列または最小限の構造で出力し、エラーにしない。・必要に応じて「規定到達者0人」をログに出す。 |

## Phase 3: 入力・出力ディレクトリの統一と検証

### 目的

規定用・全員用CSVおよびランキングJSONの「どこを入力とするか」「どこに出力するか」を整理し、全年度で一貫したパスで検証・実行できるようにする。

### 作業内容

1. **ディレクトリ役割の文書化**
   - **パス一覧**: `docs/DATA_PATHS.md` に「正」のディレクトリ一覧・スクリプト別入出力・推奨実行順序・検証手順を記載。
   - `_data/master_csv__import_1950_2024`: スクレイピング／インポート済みの**生CSV**（年度・リーグ別）。2025年もここに保存するように変更済み。
   - `_data/master_csv_calculated`: 指標計算済みの**全員用CSV**および**規定到達版CSV**（Phase 1 で生成）の格納先。
   - `public/data/rankings/`: ランキング用**JSON**の出力先（年度／リーグ／指標別）。
2. **compute_metrics_all_seasons.py の入力**
   - 現状は `_data/master_csv` を参照。`master_csv__import_1950_2024` を参照する場合は、オプション引数 `--input-dir` を追加するか、シンボリックリンク／コピーで `master_csv` に揃える運用をドキュメントに書く。
3. **検証用チェック**
   - サンプル年度（例: 1975 PL, 2024 CL）で Phase 1 → Phase 2 を実行し、規定必須指標のJSONが「規定用CSV由来」と「全員用CSV＋minPA 由来」で行数・上位が一致するか確認する。
   - 規定用CSVを意図的に削除した状態でビルドし、従来どおりの挙動（全員用CSV＋minPA）でJSONが生成されることを確認する。

### 想定ネガティブポイントと対処法

| ネガティブポイント | 対処法 |
|--------------------|--------|
| **compute_metrics が master_csv のみ読むため、import フォルダの2025が計算されない** | ・`compute_metrics_all_seasons.py` に `--input-dir` を追加し、`_data/master_csv__import_1950_2024` を指定できるようにする。・または「2025年用は手動で master_csv にコピーしてから compute_metrics を実行」と手順書に書く。・CI／ローカルで「import → compute_metrics → create_qualifying → build_rankings」の順を明示したスクリプトや README を用意する。 |
| **複数スクリプトが異なるデフォルトパスを参照している** | ・計画書と README に「正」のパス一覧を表でまとめる。・各スクリプトの先頭コメントに「入力: 〇〇 / 出力: △△」と書く。・将来的に設定ファイル（例: paths.json）に集約する余地をコメントで残す。 |
| **検証で年度ごとの規定打席値の差を人手で確認するのは大変** | ・規定打席は `qualifying_rules` + `games_map` から算出しているため、1年度1リーグだけ「算出した minPA と規定用CSVの行数」をログに出すオプション（例: `--verbose-threshold`）を Phase 1 スクリプトに追加する。・既存の `validate_qualifying_pa_table.py` や `get_min_pa_by_year.py` の結果と突き合わせられるようにする。 |

## Phase 4: サイト側の規定フィルタ整理と後方互換

### 目的

ランキング用JSONが規定必須指標について「規定到達者のみ」で既にビルドされている前提で、アプリ側の規定打席フィルタを**二重適用にならないよう**整理する。一方で、規定用CSVをまだ作っていない年度や、従来ビルドのJSONが残っている環境では、従来どおりフィルタが効くようにする。

**実施内容の詳細**: `docs/ranking_qualifying_filter_phase4.md` に規定フィルタの適用箇所・後方互換の意図を記載。

### 作業内容

1. **lib/ranking の規定フィルタ確認**
   - `adapter.ts` や `leaders.ts` 等で、ランキング表示時に「PA >= minPA」でフィルタしている箇所を特定する。
   - 規定必須指標用のJSONが「既に規定到達者のみ」で生成されている場合、フィルタを通しても結果は変わらない（全行が規定以上）ため、**現状のままでも動作上は問題ない**。
   - オプション: JSON にメタ情報（例: `"qualifyingFilterApplied": true`）を付け、アプリ側で「既に規定適用済み」ならフィルタをスキップする実装にすると、意図が明確になる。ただし Phase 4 では必須としない。
2. **後方互換**
   - 古いJSON（規定フィルタ未適用で全員入り）を読み込んだ場合でも、現行の規定フィルタが効けば表示は正しくなる。つまり「フィルタを残す」ことで後方互換を確保する。
   - 規定用CSVを導入した年度からは「規定あり版JSON = 規定到達者のみ」となり、フィルタは実質 no-op になる。

### 想定ネガティブポイントと対処法

| ネガティブポイント | 対処法 |
|--------------------|--------|
| **「規定適用済み」かどうかがJSONから分からない** | ・Phase 4 では、従来どおりクライアント／アダプタで規定フィルタをかける実装を維持する。・規定用CSV由来のJSONでは全員が規定以上なので、フィルタ後も結果は同じ。・将来、JSON に `qualifyingFilterApplied` 等のフラグを付ける場合は、その時点で「フラグが true ならフィルタをスキップ」とする。 |
| **一部年度だけ規定用CSVをやらずにビルドすると、規定あり版に未到達者が混ざる** | ・ビルド手順を「規定用CSVを生成してからランキングビルド」と明文化する。・ビルドスクリプト側で「規定必須指標で規定用CSVが無い場合は従来ロジック（全員用＋minPA）」にフォールバックしているため、規定用CSVを忘れてもランキングは正しく出る。・運用チェックリストに「規定用CSV生成 → ランキングビルド」の順を入れる。 |
| **パフォーマンス測定で「フィルタを外す」と速くなるか確認したい** | ・規定適用済みJSONではフィルタはほぼ no-op のため、外しても体感差は小さい可能性が高い。・必要なら開発時のみ、メタフラグでフィルタスキップを有効にするオプションを残す。 |

## Phase 5: ドキュメント・運用手順の整備

### 目的

全年度で「規定用・全員用CSV分割」を運用するための実行順序、対象スクリプト、トラブル時の確認項目を文書化する。

### 作業内容

1. **実行順序の明文化**
   - 新規年度を追加する場合: スクレイピング／インポート → `compute_metrics_all_seasons.py` → `create_qualifying_csv_all_years.py`（当該年度） → ランキングビルド（build_rankings_from_calculated または 2025用スクリプト）。
   - 既存年度を再ビルドする場合: 必要に応じて `create_qualifying_csv_all_years.py` を当該範囲で実行 → ランキングビルド。
2. **分割実行の例**
   - 「まず 1950–1959 の規定用CSVだけ生成」「次に 1960–1969」のように、`--year` を範囲指定できるようにする（または複数回 `--year YYYY` で実行）。
   - ランキングビルドも `--year` / `--league` で範囲指定できる場合は、その例をドキュメントに書く。
3. **本計画書への参照**
   - README や `docs/RANKING_PAGE_INFO.md` 等に「規定用・全員用CSVの役割と全年度展開は `docs/ranking_qualifying_csv_all_years_plan.md` を参照」と追記する。

### 初回セットアップ／年度追加時の手順（1ページまとめ）

| 目的 | 手順 |
|------|------|
| **新規年度を追加する** | 1) スクレイピング／インポート → 生CSVを `_data/master_csv__import_1950_2024/` に配置<br>2) `python scripts/compute_metrics_all_seasons.py`（必要なら `--input-dir _data/master_csv__import_1950_2024`）<br>3) `python scripts/create_qualifying_csv_all_years.py --year YYYY`（当該年度のみ）<br>4) ランキングビルド: 2024年以前は `build_rankings_from_calculated.py`、2025年は `build_rankings_2025_PL_full.py` |
| **既存年度を再ビルドする** | 1) 必要に応じて `create_qualifying_csv_all_years.py --year YYYY`（または対象年度を複数回）<br>2) `build_rankings_from_calculated.py`（`--year` / `--league` で範囲指定可）または 2025用スクリプト |
| **規定ルールや games_map を変更した** | 規定打席の算出ロジック（`scripts/qualifying/qualifying_rules.py` または `config/games_per_team_by_season.json`）を変更したら、**規定用CSVの再生成**と**ランキングの再ビルド**を行う。対象確認: `create_qualifying_csv_all_years.py --dry-run`。 |

パス一覧・スクリプト入出力: `docs/DATA_PATHS.md`。

### 想定ネガティブポイントと対処法

| ネガティブポイント | 対処法 |
|--------------------|--------|
| **手順を忘れて規定用CSVを生成せずにビルドしてしまう** | ・ランキングビルドは「規定用CSVが無ければ従来ロジック」のため、結果は正しい。・「推奨フロー」として「規定用CSV生成 → ビルド」をチェックリスト化し、CI やローカル用のラッパースクリプトで順序を固定する。 |
| **新規参加者がどのスクリプトをいつ実行するか分からない** | ・本計画書の Phase 5 に「初回セットアップ／年度追加時の手順」を1ページにまとめたサブセクションを設ける。・RUN.md や scripts/README に「ランキング用データの更新」の流れを簡潔に書く。 |
| **規定ルールや games_map の更新時に、規定用CSVを再生成すべきか判断しづらい** | ・「規定打席の算出ロジック（qualifying_rules / games_map）を変更したら、規定用CSVの再生成とランキングの再ビルドを行う」と計画書に明記する。・必要なら `create_qualifying_csv_all_years.py --dry-run` で対象ファイル一覧を確認してから一括再生成する手順を書く。 |

## Phase 実行順序のまとめ

| Phase | 内容 | 依存 |
|-------|------|------|
| 1 | 規定打席到達版CSV生成スクリプトの全年度対応 | 既存 qualifying_rules / games_map / filename_parser |
| 2 | ランキングビルドでの規定用CSV参照（全年度） | Phase 1 で生成した qualifying CSV（任意年度から開始可） |
| 3 | 入力・出力ディレクトリの統一と検証 | Phase 1–2 の動作確認 |
| 4 | サイト側の規定フィルタ整理と後方互換 | Phase 2 でビルドしたJSONの配布 |
| 5 | ドキュメント・運用手順の整備 | 上記すべての前提として並行して進め可能 |

分割実行する場合は、**Phase 1 を年度範囲を絞って実行 → Phase 2 を同じ範囲で実行 → 検証** を繰り返し、問題がなければ範囲を広げる形で進めると安全です。
