## Phase 1 評価

### 達成内容

- **全年度対応の規定打席到達版CSV生成**  
  `scripts/create_qualifying_csv_all_years.py` を実装し、`_data/master_csv_calculated/` 内の `batting_*_from_master.csv` から `*_qualifying.csv` を出力。
- **規定ルールの統一**  
  `qualifying_rules.py` の `get_qualifying_rule` / `calc_qual_threshold` / `is_player_qualified` を利用。1951/1952年PLのチーム別しきい値はスクリプト内マッピング（現在のチーム名）で対応。
- **対象範囲**  
  PRE 春秋（`batting_YYYY_PRE_spring/fall_from_master.csv`）、通常年度・CL/PL を対象。1938年 PRE は規定ルール未定義のためスキップ（計画どおり）。
- **実行結果**  
  167 ファイル中 166 件の qualifying CSV を生成（42,275 行 → 5,283 行）。命名規則は `*_from_master.csv` → `*_qualifying.csv` で統一。

### 良い点

- 計画の「まず全列で出力し、のちに列を絞る」方針に沿い、現状は全列コピーで動作。列の絞り込みは将来対応可能。
- `--year` / `--league` / `--dry-run` により範囲指定・事前確認が可能。
- 試合数マップ未整備時は CSV の G 列から試合数を推定するフォールバックを実装。

### 改善・注意点

- **列の絞り込み**  
  計画上の「規定用CSVは必要な列のみ」は未実施。現状の全列のままでも動作・サイズ面で大きな問題はないが、計画完遂の観点では Phase 1 の延長で列絞りを検討できる。
- **1938 PRE**  
  `qualifying_rules.py` に 1938 年 PRE の規定が無いためスキップ。必要ならルール追加と再実行が必要。
- **games_per_team_by_season.json**  
  未作成のため試合数は G 列推定。正確な規定打席で揃えたい場合は `build_games_per_team_map.py` 実行後に Phase 1 を再実行する運用を推奨。

## Phase 2 評価

### 達成内容

- **build_rankings_from_calculated.py**  
  規定必須指標（`METRICS_REQUIRE_QUALIFYING_PA_BY_NAME`）について、同一ディレクトリに `*_qualifying.csv` があればそれを読み `min_pa=0` で規定あり版 JSON を生成。無ければ従来どおり全員用CSV＋minPA。カウント系・規定なし版（`*_all.json`）は従来どおり全員用CSVのみ使用。
- **build_rankings_all_from_yearly_dir.py**  
  上記と同様に規定用CSVの有無を確認し、規定必須指標で規定用CSVがあればそれを入力に使用。
- **build_rankings_2025_PL_full.py**  
  変更なし（もともと規定用CSVを参照済み）。Phase 1 で 2025 の qualifying CSV が存在する前提で問題なし。
- **検証**  
  2024年 CL で規定用CSV読み込み・規定必須指標 24 件出力を確認。規定あり版は規定用CSV由来、カウント系は全員用CSV由来で整合。

### 良い点

- 規定用CSVの有無は「ファイル存在チェック」のみで分岐しており、規定用が無い年度は自動で従来ロジックにフォールバック。
- 規定あり版と規定なし版・規定到達者版（`*__qualified.json`）の役割を維持したまま、規定あり版の入力だけを切り替えている。

### 改善・注意点

- **規定用CSVが空（0 行）のとき**  
  計画では「規定あり版JSONは空配列または最小限の構造で出力し、エラーにしない」とある。現状は `write_csv` が行 0 のとき書き出しをスキップしているため、規定あり版 JSON は「前回ビルドの残り」のままになる可能性がある。規定到達者 0 人の年度で空JSONを明示的に出力するか、ビルド側で「規定用CSV行数 0 のときは空配列で上書き」する処理があるとより計画に沿う。
- **PRE_spring/fall のファイル名**  
  `batting_YYYY_PRE_spring_from_master.csv` → `batting_YYYY_PRE_spring_qualifying.csv` の組み立ては Phase 1 と Phase 2 で一致しており、現状の実装で問題なし。

## Phase 3 評価

### 達成内容

- **ディレクトリ役割の文書化**  
  `docs/DATA_PATHS.md` を新規作成。ディレクトリ役割表・スクリプト別入出力・推奨実行順序・検証手順を記載。計画書 Phase 3 から本ドキュメントへの参照を追加。
- **compute_metrics_all_seasons.py の入力**  
  `--input-dir` を追加。`_data/master_csv__import_1950_2024` 等を指定可能。相対パスはプロジェクトルート基準で解決。
- **検証用オプション**  
  `create_qualifying_csv_all_years.py` に `--verbose-threshold` を追加。規定しきい値（minPA 等）と規定到達者数をログ出力し、`qualifying_rules` / `games_map` との突き合わせが可能。
- **計画書との対応**  
  Phase 3 の「ディレクトリ役割の文書化」に DATA_PATHS.md を正のパス一覧として明記。

### 良い点

- パスとスクリプトの役割が一箇所にまとまり、新規参加者や運用時の参照がしやすい。
- import フォルダを入力にしたい場合の手順（`--input-dir`）が文書と実装の両方で一致している。
- 検証時には `--verbose-threshold` でしきい値と件数を確認できる。

### 改善・注意点

- **検証の実行**  
  計画の「サンプル年度（1975 PL, 2024 CL）で Phase 1 → Phase 2 を実行し、規定必須指標のJSONが規定用CSV由来と一致するか」「規定用CSVを削除してフォールバックするか」は手順として DATA_PATHS.md に書いたが、実際に手動で実行して結果を記録するまでは Phase 3 の「検証」としては未実施。必要に応じて実施・結果をドキュメントに追記するとよい。
- **compute_metrics の絵文字**  
  Phase 3 で起動直後の print のみ絵文字をやめたが、他に絵文字の print が残っており、Windows cp932 では途中で UnicodeEncodeError になる可能性がある。UTF-8 出力にするか、残りも ASCII 表記に揃えると安心。

## 残リスク・推奨アクション

| リスク | 影響 | 推奨 |
|--------|------|------|
| 規定用CSV未生成の年度が混在 | その年度だけ規定あり版が全員用CSV＋minPA になり、他年度とデータソースが異なる | 運用で「規定用CSV生成 → ランキングビルド」の順を徹底。DATA_PATHS.md の手順で統一。 |
| Windows での絵文字起因のエラー | 一部スクリプトが cp932 環境で落ちる | 必要に応じて `PYTHONIOENCODING=utf-8` の利用、または該当スクリプトの print を ASCII 表記に統一。 |
| 規定用CSV 0 行時の規定あり版JSON | 空の規定用CSVのとき、規定あり版が更新されず古いJSONが残る可能性 | ビルド側で「規定用CSV行数 0 なら規定あり版を空配列で上書き」する処理の追加を検討。 |
| games_map 未整備 | 規定打席が G 列推定になり、年度によっては公式とずれる | 必要なら `build_games_per_team_map.py` を実行し、Phase 1 を再実行。 |

---

## 結論

- **Phase 1〜3 は計画の範囲内で完了している**といってよい。規定用CSVの全年度生成、ビルドでの規定用CSV参照、パスと実行順序の文書化まで一貫して実装されている。
- サイト頁への反映（規定必須指標が軽量な規定到達者のみのJSONで表示される）に必要なデータパイプラインは整っており、Phase 4（サイト側フィルタ整理）・Phase 5（運用手順の明文化）に進む土台はできている。
- 上記の「改善・注意点」「残リスク」は、運用しながら必要に応じて対応すればよいレベルであり、Phase 1〜3 の評価としては**おおむね良好**と判断できる。
