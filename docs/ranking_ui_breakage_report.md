# 成績読み込み時のUI不具合 調査報告書

## 概要

ランキングページで「成績を読み込んだ途端にUIが変わる」事象について、成績読み込みとUI不具合の関係を整理し、再発防止・修正の指針をまとめる。

**報告された事象**
- 順位列と選手列の間に隙間ができる
- 英字名前（ローマ字名）が表示されなくなる
- 以前も同様の事象があった

---

## 0. 原因究明（過去の記録を踏まえた整理）

### 0.1 過去の記録

**参照**: `_data/reports/野手ランキングページ実装プロセス報告書.md` の **問題29: スタイリングの崩れ（横スクロール時）**

- **当時の症状**: 横スクロール時に「順」列と「選手名」列が固定されず、**隙間が生じる**
- **当時の対策**: **sticky positioning を使用し、左ブロック（順位＋選手）を 1 つの sticky コンテナにする**
  - つまり「順」と「選手名」を **一つの塊**（`<div className="sticky left-0 z-40 flex">` 内に 30px + 90px の div）として固定する方式

### 0.2 現行実装との差異

| 項目 | 問題29で推奨された方式 | 現行 RankingUI.tsx |
|------|------------------------|---------------------|
| 構造 | 左ブロックを **1 つの sticky コンテナ**（flex）にし、その中に順・選手の div を並べる | **&lt;table&gt;** の &lt;th&gt; / &lt;td&gt; を **セル単位**で sticky にしている（順: `left: 0`、選手: `left: 30px`） |
| 単位 | 「順＋選手」が **一塊**で動く | 各セルが **別々**に sticky |

### 0.3 根本原因の整理

1. **問題29の対策が現行では別形で実装されている**  
   当時推奨された「順＋選手を 1 つの sticky コンテナ（flex）にする」方式は、**table を使わず左ブロックだけを div で一塊にする**設計だった。現行は **table を維持したまま、各セルに `position: sticky` を付けている**ため、推奨と異なる実装になっている。

2. **table セル単位の sticky が隙間を生みやすい理由**  
   - テーブルの列幅は、thead / tbody のセル内容・border・`table-layout` などで決まる。
   - tbody が **0 行 → N 行**に変わるタイミングで、ブラウザが列幅を再計算する。
   - **隣接する 2 つの sticky セル**（順列と選手列）は、それぞれ `left: 0` と `left: 30px` で固定されるが、列幅の再計算タイミングや border の描画がセルごとになるため、**境界がずれて隙間のように見える**ことがある。
   - 横スクロール時も、セル単位で sticky が効くため、テーブル列幅と `left` の基準がずれれば同様のずれが発生しうる。

3. **再発の構造**  
   問題29では「1 塊の sticky コンテナ」で隙間を防いだが、現行は **table + セル単位 sticky** に変わっている（あるいは途中で table 構成に戻した）ため、**同じ「順・選手列の隙間」が別のトリガー（今回は成績読み込みによる 0 行→N 行）で再発している**。

### 0.4 結論

- **直接のきっかけ**: 成績読み込みにより tbody が 0 行→N 行になり、テーブルレイアウトが再計算されるタイミングで、順列・選手列の境界がずれて隙間として見える。
- **構造的な要因**: 問題29で有効だった「順＋選手を 1 つの sticky コンテナにする」方式が、現行では **table のセル単位 sticky** になっており、列幅の変動や境界の描画に弱い。
- **今回の対処（table-layout: fixed + colgroup）**: 列幅を先頭行で固定し、0 行→N 行の切り替え前後で列幅が変わらないようにすることで、**セル単位 sticky のまま**隙間を抑える狙い。根本的には、問題29と同様に「順＋選手を 1 塊の sticky コンテナ」に戻す選択肢もある（table 構造の見直しが必要）。

## 1. 成績読み込みとUIの関係性

### 1.1 データ表示フローの変化（案A 導入後）

| タイミング | 以前（CSV直接読み込み） | 現在（案A: 指標ごとJSON取得） |
|-----------|-------------------------|------------------------------|
| 初回描画 | サーバーでCSVを読み、行データに変換してからHTMLを生成。**初回から行データあり**。 | サーバーは指標リストのみ渡す。**初回は `rowsFromJson = []`（空）**。 |
| データ到着後 | （不要・初回からある） | Client で `loadRankingJson` が完了し `setRowsFromJson(data)` で**一括で行が入る**。 |

つまり **「成績読み込み」＝ Client で JSON を取得し `rowsFromJson` に set するタイミング**であり、その前後で次の変化が起きている。

1. **行数**: 0 行 → N 行（一気に増える）
2. **テーブル構造**: thead のみの状態 → thead + tbody に多数の `<tr>` ができる
3. **行オブジェクトの形**: なし → JSON のキー・値の形そのまま

この「空 → 一括でデータが入る」という**状態の切り替わり**が、レイアウトの再計算と表示内容の差として表れている。

### 1.2 関係性の整理

- **順列・選手列の隙間**  
  「成績読み込み」＝ **tbody が 0 行 → N 行になる瞬間** に、テーブルの列幅・sticky の位置が再計算され、その結果として隙間が目立つ。
- **英字名前が消える**  
  「成績読み込み」＝ **表示行が CSV 由来から JSON 由来に切り替わった** 結果、行オブジェクトのキーや値の有無が変わり、`romanName` が無い・空になるケースが発生している。

どちらも「読み込み」そのものが悪いのではなく、**読み込みによって「空のテーブル → データ入りテーブル」かつ「データソースが CSV 由来 → JSON 由来」に変わること**が原因になっている。

---

## 2. 順列・選手列の隙間

### 2.1 想定原因

1. **空 tbody とデータ入り tbody でレイアウトが変わる**
   - tbody が 0 行のときは列幅が thead の `<th>` だけで決まる。
   - データ到着後は `<td>` が大量に増え、ブラウザがテーブルレイアウトを再計算する。
   - `table-layout: fixed` が無いため、この再計算で列幅や sticky の基準が変わり、隙間として見える可能性がある。

2. **sticky 列の幅・位置のずれ**
   - 順列: `width: 30px`, `left: 0`
   - 選手列: `width: 90px`, `left: 30px`
   - レイアウト再計算のタイミングで、これらの値が効くタイミングがずれると、見た目上「隙間」になる。

3. **0 行のときのテーブル幅**
   - 行が無いときはテーブル全体の幅や列割りが、データありのときと異なる可能性がある。

### 2.2 推奨対応

- **テーブルに `table-layout: fixed` を指定する**  
  → 列幅を先頭行（thead）で固定し、データ到着前後で列幅が変わらないようにする。
- **先頭行（thead）で列幅を明示する**  
  → 順列 30px・選手列 90px など、既存の `RANK_WIDTH` / `PLAYER_WIDTH` を thead で一貫して使う（現状もほぼそうなっているが、`table-layout: fixed` とセットで効かせる）。
- （必要なら）**初回から 1 行だけスケルトン行を描画する**  
  → tbody が「0 行 → N 行」ではなく「1 行 → N 行」に変わるようにし、列幅の計算が最初からデータありに近い形で行われるようにする。

## 3. 英字名前が表示されなくなる件

### 3.1 データソースの違い

| 経路 | 英字名の取得元 | 備考 |
|------|----------------|------|
| **以前（CSV経由）** | `lib/ranking/leaders.ts` の `getPlayerNames()` | CSV の `player_name_en` / `romanName` / `name_en` などを参照。 |
| **現在（JSON経由）** | ビルドスクリプト（Python）が CSV から JSON を生成する際に `romanName` をセット | スクリプトは `player_name_en`（または同等の列）があれば `romanName` に入れる。列が無い・空なら `''`。 |

UI はどちらも **`row.romanName`** を参照している（`RankingUI.tsx` の `hasRomanName` / 表示部分）。

### 3.2 想定原因

1. **元 CSV に英字名列が無い・空**
   - 対象年度・リーグの master CSV に `player_name_en`（や同等列）が無い、または行ごとに空の場合、JSON の `romanName` は全て空になり、英字名は一切表示されない。

2. **列名の不一致**
   - Python 側は `player_name_en` などを前提にしているが、実際の CSV が `name_en` / `roman_name` など別名だと、マッピングされず `romanName` が空になる可能性がある。

3. **キー名の違い（JSON → UI）**
   - JSON が `RomanName` や `roman_name` で出している場合、UI は `row.romanName` しか見ないため表示されない。現状のビルドスクリプトは `romanName` で出力しているため、同じ年度・同じデータなら 1・2 が主因になりやすい。

### 3.3 推奨対応

- **元データの確認**
  - 対象の `batting_*_from_master.csv` に `player_name_en`（または英字名用列）が存在するか、値が入っているかを確認する。
- **JSON 生成側の統一**
  - 英字名に使う列名を 1 つに決め、複数列（`player_name_en`, `name_en`, `roman_name` など）をフォールバックで参照するようにする（必要なら Python 側の仕様として明文化）。
- **Client 側の耐性**
  - JSON の行をそのまま渡すのではなく、表示用に正規化する層を挟む。  
    - 例: `romanName = row.romanName ?? row.roman_name ?? row.RomanName ?? ''` のように複数キーを許容し、`RankingRow` の形に揃えてから `RankingUI` に渡す。  
  - これで将来キー名が変わっても、英字名が消えにくくなる。

## 4. 「以前も同様のことがあった」について

- **過去の記録**: `_data/reports/野手ランキングページ実装プロセス報告書.md` の **問題29** で、横スクロール時に「順」列と「選手名」列の隙間が生じる事象が記録され、**「順＋選手を 1 つの sticky コンテナにする」** 対策が推奨されている。今回の隙間は、**トリガーが「横スクロール」ではなく「成績読み込み（0 行→N 行）」** である点が異なるが、**「順・選手列の境界がずれて隙間になる」** という現象は同じ系列である。
- データソースの切り替え（CSV → 別CSV、または CSV → JSON）や、**「初回は空で、後からデータが流し込まれる」** ような変更のたびに、
  - テーブルが「0 行 → N 行」でレイアウトが変わる
  - 行オブジェクトの形（キー名・欠損）が変わる  
という同じパターンで、隙間や英字名非表示が再発しうる。
- そのため、**「成績が読み込まれた瞬間にUIが変わる」＝「データが入ったタイミングと、レイアウト・表示仕様の前提がずれている」** と捉えると再発防止しやすい。

## 5. 修正の優先度とタスク例

| 優先度 | 内容 | タスク例 |
|--------|------|----------|
| 高 | 順列・選手列の隙間を防ぐ | `RankingUI.tsx` の `<table>` に `style={{ tableLayout: 'fixed' }}` を追加。必要なら `<colgroup>` で順列・選手列の幅を明示。 |
| 高 | 英字名の表示を安定させる | (1) 対象 CSV の `player_name_en` 有無・中身の確認。(2) Client で JSON 行を正規化する処理を追加し、`romanName` / `roman_name` / `RomanName` などをまとめて `romanName` にマッピングしてから UI に渡す。 |
| 中 | 初回描画の安定化 | ローディング中も tbody にスケルトン行を 1 行出す、など「0 行」の時間を減らし、列幅が最初から確定するようにする。 |
| 低 | 仕様の明文化 | 「ランキング用 JSON の行は `name` / `romanName` / `playerId` / `team` / `rank` を必ず持つ」「英字名は CSV の ○○ 列を参照する」などをドキュメントに残す。 |

---

## 6. まとめ

- **成績読み込みとUI不具合の関係**
  - 読み込み = **Client で JSON が届き、`rowsFromJson` が 0 行 → N 行に変わるタイミング**。
  - そのタイミングで (1) テーブルレイアウトの再計算により順列・選手列の隙間が発生し、(2) 行データが JSON 由来になることで `romanName` の有無・キーが変わり、英字名が消えて見える。
- **直すためにやること**
  - テーブル: `table-layout: fixed` と列幅の一貫した指定で、データ到着前後でレイアウトが変わらないようにする。
  - 英字名: 元 CSV の列と JSON の `romanName` を揃え、Client では複数キーを許容して `romanName` に正規化してから UI に渡す。

この報告書を前提に、上記タスクを実施すれば、同種のUI不具合の再発を抑えやすくなる。
