# 2025年セ・パ ランキングページ改善計画書

## 目的・達成条件

以下の3条件を満たすランキングページを、**Phase（段階）** に分けて実装する。

1. **規定到達者CSV／全員版CSVの使い分け**  
   指標ごとにどちらのCSVを適用するかは、**2024年以前の仕様に合わせる**（規定必須＝規定用CSV、カウント系＝全員版CSV）。
2. **英字名前の表示**  
   ランキング行に英字名（romanName / player_name_en）が**ある状態**で表示される。
3. **2025年新規入団選手の反映**  
   「新規入団選手を更新する」形で追加し、規定・非規定どちらのランキングにも漏れなく載る。

---

## 現状の未熟な点（整理）

| 領域 | 未熟な点 |
|------|----------|
| データ | 2025年CLの全員版CSVが外部（npb_batting）由来のため、新入団選手が欠けることがある。PLは全員版が少人数のままの可能性。 |
| 規定／全員の適用 | ビルド側で規定用・全員版の切り替えは実装済みだが、2025年でJSON取得パス・正規化が2024年以前と完全に同一か要検証。 |
| 英字名 | JSONに `romanName` が無い行は API `/api/roman-names/{year}/{league}` でCSVから補完しているが、CSVに `player_name_en` が無い選手は空白のまま。 |
| 新入団の追加 | 報告書ベースのマージスクリプトはあるが、運用フロー（報告書の更新タイミング・バックアップの扱い）が明文化されていない。 |

## Phase 一覧とネガティブポイント／解決策

### Phase 1: 規定用CSV・全員版CSVの適用ルールの明確化と検証（2024年以前と同じ）

**内容**

- 指標ごとに「規定用CSV」か「全員版CSV」を使うかを、2024年以前のランキングビルド・表示ロジックと一致させる。
- 規定必須指標: `*_qualifying.csv` から生成したJSONを表示用に使用（`{metric}.json` は規定到達者のみ）。
- カウント系指標: 全員版 `*_from_master.csv` から生成した `{metric}_all.json` を表示用に使用（Client は `useAllPlayers=true` で `_all.json` を取得）。
- ビルド後の `normalize_metric_json_for_ui` により、UIが常に `{metric}.json` を読む場合でも、Bグループ（規定不要）は `.json` が `_all.json` と同一内容になるよう正規化済みであることを確認する。

**想定ネガティブポイントと解決策**

| ネガティブポイント | 解決策 |
|--------------------|--------|
| 2025年のみ規定ルール（試合数・規定打席）が2024年以前と異なる可能性がある | `lib/ranking/qualifyingPA.ts` の `calculateMinPA` および `scripts/qualifying/qualifying_rules.py` に2025年を明示し、同一ロジックで規定打席を算出する。必要なら `config/games_per_team_by_season.json` を整備する。 |
| ビルド側の指標名（Record.csv／metric_map）とフロントの `qualifyingPA.ts` の Set が食い違うと、規定／全員の切り替えがずれる | 指標名の正規化（BB/K→bbk、打率→avg 等）を一箇所にまとめ、ビルド・フロント両方で同じ分類になるようドキュメントと単体チェックを追加する。 |
| 既存JSONが「全員入り」のまま残っている年度があり、規定必須指標で規定未到達者が表示される | Phase 4 と同様、Client 側の規定フィルタ（PA >= minPA）を残し、従来JSONでも規定未到達者を除外する。規定用CSV由来のJSONではフィルタは実質 no-op。 |

**Phase 1 実施メモ（実施済み）**

- **2025年規定ルール・試合数**: `qualifying_rules.py` で CL/PL とも `1959 <= year <= 2025` で PA_TEAMGAMES_MULT（3.1倍）を適用。`qualifyingPA.ts` の TEAM_GAMES_DATA に 2025（CL: 143, PL: 143）を明示。2024年以前と同じルールで踏襲。
- **試合数マップ**: `config/games_per_team_by_season.json` を新規作成。2020–2025 の CL/PL 試合数を記載（2024・2025 は 143）。ビルド時に「試合数マップが見つかりません」のフォールバックを解消。
- **規定／全員の適用**: ビルド側は規定必須指標で `*_qualifying.csv`、カウント系で全員版CSVを使用し、`normalize_metric_json_for_ui` で B グループは `.json` を `_all.json` と同一化済み。Client は `shouldRequireQualifyingPA(metricDef.key)` に応じて `{metric}.json` または `{metric}_all.json` を取得。Client 側の規定フィルタ（PA >= minPA）は残してあり、従来JSONでも規定未到達者を除外（`docs/ranking_qualifying_filter_phase4.md` の通り）。

### Phase 2: 全員版CSVの整備と2025年新入団選手の「更新」フロー確立

**内容**

- 2025年CL・PLの**全員版CSV**（`batting_2025_*_from_master.csv`）を、規定用CSVの母集団としても一貫して参照できるように整備する。
- 2025年新入団選手は「**報告書（2025_new_players_report.csv）を更新し、マージスクリプトで全員版に追加する**」形に統一する。
- マージ対象: 報告書に載っているが全員版CSVにいない選手。バックアップに成績があればそれを流用、無ければ最小行（名前・チーム・年・リーグのみ）を追加し、非規定ランキングに名前が載るようにする。

**想定ネガティブポイントと解決策**

| ネガティブポイント | 解決策 |
|--------------------|--------|
| 報告書に載せ忘れた新入団選手がずっとランキングに出ない | 運用ルールとして「シーズン中・オフの新入団判明時に報告書に1行追加し、`merge_2025_new_players_from_report.py` を実行する」を明文化する。必要なら報告書の項目に「備考・出典」を追加し、後から追記しやすくする。 |
| バックアップにいない新入団選手は最小行（数値0）になり、非規定ランキングで「上位100件」に含まれず画面に出ない | 現状は「名前は全員版CSVに存在する」ことを優先。実成績は別ソース（手動CSV・他システム）が用意できた時点で全員版CSVの該当行を差し替え、ランキングを再ビルドする。必要なら「新入団のみ一覧表示」などの別UIを検討する。 |
| CLとPLで全員版CSVの列形式（npb_batting と TopPage）が違うため、マージスクリプトが複雑になる | 既存の `merge_2025_new_players_from_report.py` でCLはバックアップ→現在形式のマッピング、PLは同一形式コピーで対応済み。新リーグ・新形式を足す場合は「チーム→リーグ」と「バックアップ→現在」のマッピング表を1ファイルにまとめ、スクリプトから参照する。 |

**Phase 2 実施メモ（実施済み）**

- **全員版CSVの整備**: 2025年CL・PLの `batting_2025_*_from_master.csv` を規定用CSVの母集団として一貫参照。報告書ベースのマージを実行し、報告書の全選手が全員版CSVに含まれることを確認（追加件数0＝既に同期済み）。
- **運用ルールの明文化**: 「シーズン中・オフの新入団判明時に報告書に1行追加し、`merge_2025_new_players_from_report.py` を実行する」を `docs/DATA_PATHS.md` に「2025年ランキング更新チェックリスト」として番号付きで記載。手順は 報告書更新 → マージ → 規定用CSV作成 → ランキング再ビルド（CL/PL）。
- **報告書の項目**: `_data/reports/2025_new_players_report.csv` に「備考」列を追加し、後から出典・メモを追記しやすくした。

### Phase 3: 英字名前（romanName）の充足

**内容**

- ランキング表示行で、**英字名が空にならない**ようにする。
- 方針: (1) ランキングJSONをビルドする際に、元CSVの `player_name_en` を `romanName` として出力する。(2) 不足分は既存の `/api/roman-names/{year}/{league}` で全員版CSV（または規定用CSV）から補完する。
- 2025年新入団選手は報告書に `player_name_en` があるので、マージ時に全員版CSVに含めれば、ビルドまたはAPIで拾えるようにする。

**想定ネガティブポイントと解決策**

| ネガティブポイント | 解決策 |
|--------------------|--------|
| ビルドスクリプトが現在、JSONに `romanName` を出していない | `build_rankings_from_calculated.py`（および 2025_PL_full）のランキング行生成で、CSVの `player_name_en` を `romanName` として含める。既存の `name_en` 等とのキー揺れは、Client の `normalizeRankingRow` で吸収済みなので、キー名を `romanName` に統一する。 |
| 全員版CSVに `player_name_en` が無い行がある（欠損・空白） | 報告書や他CSVで英字名が分かれば、全員版CSVの該当行を更新する。APIは「名前|チーム」でマッチするため、CSVに1度でも入れば補完される。欠損が残る選手は「英字名なし」のまま表示し、後からCSVを更新して再ビルドで解消する。 |
| 同一選手が複数表記（J.オスナ／オスナ）で別行になり、英字名マップが片方にしか効かない | 報告書・マージ時に「同一選手の表記ゆれ」を1つにそろえる。表示側は `normalizeRankingRow` で `name` と `romanName` を扱うだけにし、データ側で正規化する。 |

**Phase 3 実施メモ（実施済み）**

- **既存選手（2024年以前に名前がある選手）**: 2024年以前の `*_from_master.csv` から `player_name_ja`＋`team` をキーに英字名を収集し、2025年全員版CSVの **空欄行のみ** に流用。`scripts/phase3_fill_roman_names_2025.py` の Step 1 で実行。
- **新選手（`docs/2025_new_players_names_only.md` に名前があるメンバーのみ）**  
  - **日本人**: NPB公式の選手個人ページ（`id=pc_v_kana`）からひらがなをスクレイピングし、ヘボン式でローマ字変換して `player_name_en` に反映。報告書の `name_kana` も更新。  
  - **新外国人**: 表示が独特な例があるため、前回の表示形式を踏襲。**名前の下にスペルを表示する3例**は以下とする。  
    1. **キャベッジ** → 英字表記 `Tyler Cabbage`（表示: 名前「キャベッジ」の下に `T. Cabbage`）  
    2. **ファビアン** → 英字表記 `Sandro Fabian`（表示: 名前「ファビアン」の下に `S. Fabian`）  
    3. **ボスラー** → 英字表記 `Bosler`（単一表記のためそのまま、名前の下に `Bosler`）  
  上記は `scripts/phase3_fill_roman_names_2025.py` 内の `FOREIGN_ROMAN_LOOKUP` で定義し、報告書・全員版CSVの `player_name_en` に設定。ビルド時に `romanName` としてJSONに出力され、`RankingUI` の「名前の下にスペル」表示でそのまま利用される。
- **実行**: `python scripts/phase3_fill_roman_names_2025.py`（オプション: `--skip-scrape` でスクレイピング省略、`--dry-run` で流用のみ確認）。実行後、規定用CSV再生成・ランキング再ビルドでJSONに反映される。

### Phase 4: ランキング再ビルドと表示の一貫性確認

**内容**

- Phase 1〜3 の変更を入れたうえで、2025年CL・PLの規定用CSV作成 → 全員版CSV更新（新入団マージ）→ ランキングJSON再ビルド → サイト表示、の一連の流れを実行する。
- 規定必須指標では規定到達者のみ、カウント系では全員版（新入団含む）が表示されること、および英字名ができるだけ表示されることを確認する。

**想定ネガティブポイントと解決策**

| ネガティブポイント | 解決策 |
|--------------------|--------|
| 再ビルド後にキャッシュの影響で古いJSONが返る | プロキシやCDNのキャッシュ無効化、または `revalidate=0` 等の既存設定を確認する。開発時はハードリロードで確認。 |
| 非規定指標で「100件」に切っているため、新入団選手が下位だと一覧に登場しない | 現状仕様として許容する。必要なら「表示件数」を増やすか、新入団フラグを付けて「新入団のみ表示」を追加する検討は Phase 5 以降とする。 |
| 規定用CSVと全員版CSVの選手数が年度によって大きく違うと、ユーザーが混乱する | ヘルプやページ内説明で「規定打席到達者」と「全選手」の切り替えが指標によって変わることを一文で明記する。 |

**Phase 4 実施メモ（実施済み）**

- 報告書の英字名を from_master に反映（`apply_report_en_to_from_master_2025.py`）→ 規定用CSV再生成（`create_qualifying_csv_2025.py`）→ ランキングJSON再ビルド（`build_rankings_2025_PL_full.py --year 2025 --league CL/PL`）を実行。規定必須指標・非規定指標とも英字名・新入団選手が表示されることを確認済み。

## 計画書を作るにあたって生まれた疑問（Chatで確認したいこと）

計画書をより良くするため、以下をChatで確認・議論するとよい。

### データ・運用

1. **2025年全員版CSVの「正」はどこか**  
   現在、CLは npb_batting 由来の全選手CSV＋報告書マージ、PLは別ソースの22人など、リーグごとに出自が違う。2025年について「これが正の全員版」と定める一次ソース（npb_batting / 手動CSV / 他システム）を決めてよいか。

2. **新入団選手の「成績」の一次ソース**  
   バックアップにいない新入団選手（例: チェイビス）は、現状は最小行（0）で追加している。実成績を入れる場合、公式記録・他プロジェクトのCSV・手入力のどれを一次ソースとするか、運用ルールを決めてよいか。

3. **非規定ランキングの表示件数**  
   非規定指標は「上位100件」で切っているため、新入団で成績が少ない選手は一覧に出ない。100件のまま運用するか、表示件数拡張や「新入団のみ」フィルタを検討するか。

### 規定・指標

4. **2025年の規定打席・試合数の定義**  
   サイト側は `qualifyingPA.ts` の `calculateMinPA`、ビルド側は `qualifying_rules.py` と `games_per_team_by_season.json`（またはフォールバック）を参照している。2025年CL・PLの試合数・規定打席を「2024年以前と同じ算出方法」で揃えるか、2025年専用の値を許容するか。

5. **指標の追加・名前変更時の同期**  
   新指標を足すとき、`Record.csv`・`metric_map.json`・`qualifyingPA.ts` の Set・ビルド側の METRICS_REQUIRE/NO_QUALIFYING の4箇所を揃える必要がある。正を1つ（例: Record.csv）にし、他をコード生成やチェックスクリプトで揃える運用にしてもよいか。

### 英字名

6. **英字名の「正」**  
   英字名は CSV の `player_name_en`、報告書の `player_name_en`、API の補完の3経路がある。優先順位（例: ランキングJSONの出力 > API補完 > 空白）を明文化してよいか。

7. **表記ゆれ（J.オスナ／オスナなど）**  
   同一選手の日本語名・英字名の表記ゆれを、データ側でいつ正規化するか（報告書更新時・マージ時・ビルド時）。表示側は「受け取った名前をそのまま表示」でよいか。

### スコープ・優先度

8. **2025年のみを対象とするか**  
   本計画は「2025年セ・パ」に焦点を当てている。2024年以前のランキングページの挙動・データは変えず、2025年だけ Phase を適用するか。それとも全年度で規定／全員／英字名のルールをそろえるか。

9. **Phase の実施順序の固定**  
   上記は Phase 1→5 の順を想定している。英字名（Phase 3）を先にやり、そのあと規定／全員（Phase 1）を検証するなど、順序を変えてもよいか。
