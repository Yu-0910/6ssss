# 日本語パス問題の原因と対処

プロジェクトが **OneDrive\ドキュメント\デスクトップ\TopPage** のように日本語（「ドキュメント」）を含むパスにあるとき、**Cursor のターミナルから `cd` や `Set-Location` でそのパスを指定すると失敗する**現象の原因と対処をまとめる。

---

## 1. 現象

Cursor で「ターミナルでコマンドを実行」すると、次のようなエラーになる。

```
Set-Location :pX 'C:\Users\short\OneDrive\ドキュメンチEチEクトップ\TopPage' ݂Ȃߌoł܂B
指定されたパス 'C:\Users\short\OneDrive\ドキュメンチEチEクトップ\TopPage' が見つからないため、検出できません。
```

- **期待するパス**: `C:\Users\short\OneDrive\ドキュメント\デスクトップ\TopPage`（「ドキュメント」）
- **実際に渡されているパス**: `ドキュメンチEチEクトップ` のように **「ドキュメント」が文字化け** している

そのため、`指定されたパスが見つかりません` となる。

---

## 2. 原因の整理

### 2.1 どこで化けているか

1. **Cursor がコマンドを実行する流れ**
   - 実行時に「作業ディレクトリ」（ワークスペースのパス）を **文字列として** ターミナルに渡す。
   - その際、**一時的な PowerShell スクリプト**（`ps-script-xxxx.ps1`）を生成し、その中で  
     `Set-Location 'C:\Users\short\OneDrive\ドキュメント\デスクトップ\TopPage'`  
     のような形でパスが埋め込まれる。

2. **文字化けが起きるポイント**
   - ワークスペースパスは **UTF-8** で「ドキュメント」が含まれた状態で保持されていると考えられる。
   - 一方で、**PowerShell がそのスクリプトを読むときのエンコーディング**、または **スクリプトを書き出すときのエンコーディング**が UTF-8 と一致していない可能性がある。
   - 日本語 Windows ではコンソールの既定コードページが **CP932（Shift_JIS）** のことが多い。
   - その結果、**UTF-8 のバイト列が CP932 として解釈**され、「ドキュメント」が「ドキュメンチEチEク」のような別の文字列に化ける。
   - 化けた文字列は実際のフォルダ名と一致しないため、`Set-Location` が「パスが見つからない」と失敗する。

つまり、

- **原因**: Cursor から渡される「日本語を含むパス」が、**PowerShell 実行時のエンコーディング（UTF-8 / CP932 の食い違い）** により文字化けしている。
- **直接のきっかけ**: 一時スクリプトに埋め込まれたパス文字列の **渡し方・保存時のエンコーディング** と、**PowerShell が読むときのエンコーディング** が一致していない。

### 2.2 既知の類似事象

- **PowerShell** では、日本語の句点「。」などがパスに含まれると、ドラッグ＆ドロップやコピーで **文字が落ちる／化ける** 不具合が報告されている（[PowerShell #14949](https://github.com/PowerShell/PowerShell/issues/14949)）。
- 同種の事象は **Windows Terminal** 側の課題としても言及されている（[terminal #9550](https://github.com/microsoft/terminal/issues/9550)）。
- いずれも **「ツールがパス文字列をどう受け渡し・どう解釈するか」** と **コンソール／スクリプトのエンコーディング** が絡んでいる。

今回の「ドキュメント」の化けも、**同じ種類の「非ASCIIパスの受け渡し・解釈」の問題**とみなせる。

### 2.3 バッチから実行すると動く理由

- **エクスプローラーで `run_phase1b_2000_2003_via_all_players.bat` などをダブルクリック** した場合は、**Windows がその .bat のフルパスを OS レベルで解決**してから cmd.exe を起動する。
- バッチ内では **`cd /d "%~dp0"`** だけを使っている。
  - `%~dp0` は **「このバッチがあるディレクトリ」** を、**OS が持っている正しいパス** として cmd に渡す。
  - ここでは **Cursor がパス文字列をスクリプトに埋め込んでいない** ため、**エンコーディングの変換や解釈のずれが発生しない**。
- そのため、**同じ日本語パスでも、バッチ経由なら正しく `cd` でき、その後の `python scripts\...` も問題なく動く**。

---

## 3. 対処法

| 対処 | 内容 |
|------|------|
| **推奨: バッチをエクスプローラーから実行** | `run_phase1b_2000_2003_via_all_players.bat` などを **エクスプローラーでダブルクリック**して実行する。`%~dp0` により日本語パスでも正しくプロジェクト直下に移動し、その後に Python が動く。 |
| **Cursor のターミナルを「既にプロジェクトで開いた状態」で使う** | Cursor でプロジェクトを開いたときに、**最初からカレントディレクトリがプロジェクトルート**になっている場合、**追加の `cd` / `Set-Location` をしない**で `python scripts\...` だけ実行すると、パスを渡さずに済むため化けない。 |
| **8.3 短いパスを使う（ある場合）** | `dir /x` で短いパス（例: `DOCUME~1`）を確認し、そのパスで `cd` する方法。環境によっては短いパスが無い、または同じ問題が起きる場合がある。 |
| **ターミナルのコードページを UTF-8 に合わせる** | ターミナルで先に `chcp 65001` を実行してから、同じターミナルでコマンドを実行する。**Cursor が「スクリプトを書き出す／渡す」段階で既に化けている場合は、これだけでは解消しない**可能性がある。 |

---

## 4. まとめ

- **原因**: Cursor が作業ディレクトリのパスを PowerShell に渡す過程で、**UTF-8 と CP932（など）の解釈のずれ** により「ドキュメント」が文字化けし、存在しないパスとして `Set-Location` が失敗している。
- **再現条件**: ワークスペースパスに日本語（ここでは「ドキュメント」）が含まれており、Cursor がそのパスを **一時 PowerShell スクリプトに埋め込んで** `Set-Location` している場合。
- **確実な対処**: **日本語を含むパスを Cursor 経由で渡さない**こと。具体的には **.bat をエクスプローラーから実行** し、`%~dp0` でプロジェクト直下に移動してから Python を実行する。

計画書の「2004年形式で 2003→2000 を実行」は、**`run_phase1b_2000_2003_via_all_players.bat` をエクスプローラーから実行** すれば、日本語パス環境でも実行できる。
