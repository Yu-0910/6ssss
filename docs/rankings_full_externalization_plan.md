# ランキングデータ本番移行 全年度外部参照 計画書

**目的**: 本番移行として、Cloudflare R2 に `public/data/rankings/` 配下の**全年度・全リーグ・全指標**の集計済みランキングデータ（JSON；CSV が存在すれば含める）のみを、**同一ディレクトリ構造**で配置する。Next.js のページ・API・lib・計算ロジック・output/backups などの生成物やバックアップ類は一切移行せず、参照は既存の `/data/rankings/*` プロキシ経由に統一したまま、**全年度の外部参照を有効化**する。

**前提**:
- R2 バケット・公開URL は既に用意済み（`docs/cloudflare_r2_vercel_setup_plan.md` の Phase 1 済み）。プロキシルート（`app/data/rankings/[...path]/route.ts`）と URL ヘルパー（`lib/ranking/url.ts`）は既存のまま利用する。
- **Vercel にはランキングデータを載せない**: `public/data/rankings/` は約 879MB 級のため、Vercel への一括デプロイは重く不可能。そのため **.vercelignore で除外**し、ランキングは **R2 から取得**する前提で本計画を組んでいる。Vercel にデプロイするのは**アプリ本体（コード）のみ**。データは R2 に置いたまま参照する。
- **サイトは既に Vercel からデプロイ済み**。Phase 1〜3（R2 へのアップロードまで）は既存プロジェクトのまま実施可能。Phase 4 は既存の Vercel 本番環境に環境変数を設定し、Redeploy する。Phase 5 の動作確認は本番URLで行う。

## 確定方針（データ構成・運用・切り替え・制限）

以下は Chat で確定した方針。Phase 実施時に参照する。

### ① データ構成

| 項目 | 方針 |
|------|------|
| **CSV の有無** | 現状は **JSON のみ移行**で進める。将来 `public/data/rankings/` に CSV を置く場合は、**同じルールで R2 にも同階層で追加**すればよい（現時点の作業は不要）。 |
| **年度・リーグのゆれ** | **「ローカルにある構造をそのまま R2 に写す」**で OK。正規化や例外処理は不要（既存の参照パスがその構造前提のため、変えると壊れる）。**「パスは真実」**: ローカルのディレクトリ構造 ＝ R2 上の構造。 |
| **RANKINGS_EXTERNALIZE_SCOPE** | **scope 空 ＝ 全年度外部参照**で問題なし。これを本番の最終形とする。**前提**: R2 側に全年度が揃っている状態で切り替える。 |

### ② アップロード・運用

| 項目 | 方針 |
|------|------|
| **アップロード認証** | まずは **手元で 1 回（ローカル手動アップロード）**でよい。CI は後回し。トークンは **ローカルの環境変数 or パスワードマネージャ**で管理し、**リポジトリには絶対に入れない**。 |
| **再アップロード方針** | 当面は **「全置換」推奨**（シンプルで事故が少ない）。運用が固まったら差分（sync）に移行を検討。**削除されたファイルは R2 からも削除**する（古い指標の残骸が参照されて混乱するため）。 |
| **R2 のバックアップ** | 本番の唯一ソースにするなら、**「別バケットへ定期コピー」**を推奨（例: rankings-data → rankings-backup）。まずは手動でも OK。安定後に自動化（Workers/CI）を検討。 |

### ③ 本番切り替え・ロールバック

| 項目 | 方針 |
|------|------|
| **.vercelignore と public/data/rankings** | **public/data/rankings/ は常にデプロイ除外**（Vercel にデータを載せた一括公開は重く不可能なため、行わない）。ロールバックは **RANKINGS_EXTERNALIZE_SCOPE の復元**や **R2 側のデータ修復**で対応する。「除外を外してデータごと再デプロイ」は行わない（重すぎる）。 |
| **段階的切り替え** | すでに動作確認済みなので **全年度一括で R2 切替**で OK。安全策として、切替後に**代表年（古い/新しい、CL/PL）だけ確認**して完了とする。 |

### ④ パフォーマンス・制限

| 項目 | 方針 |
|------|------|
| **ファイル数・サイズ** | 現状の総量は目安 **約 879MB 級**。R2 無料枠（容量 10GB 想定）の範囲に概ね収まる見込み。**移行前にローカルで `public/data/rankings` の総サイズとファイル数を一度だけ集計して確認**する（Cursor 側でコマンド実行で OK）。 |
| **レート制限（r2.dev）** | r2.dev はレート制限があり本番推奨ではない。最終的には **カスタムドメイン（例: data.example.com）を R2 に紐づける** or **現状の Vercel プロキシ＋キャッシュで吸収**する方針。いまはプロキシで Cache-Control も付いているので、**当面の本番は問題になりにくい**。 |

## Phase 1: 対象データの確定と R2 パス仕様の確認

**内容**

- 移行対象を「`public/data/rankings/` 配下の全ファイル」に限定することを明文化する。
- R2 上でのパスを「現在の URL パス（プロキシが期待するパス）と一致させる」と定義する。
- 参照は変更しないこと（ページ・API・lib・output/backups は移行しない、`/data/rankings/*` プロキシ経由のまま）を確認する。

**実施手順**

1. **対象ディレクトリ**: `public/data/rankings/` のみ。サブディレクトリは年度（例: 1936, 1950, 2025）→ リーグまたはシーズン（例: CL, PL, PRE_spring, PRE_fall）→ 指標 JSON（例: OPS.json, 打率.json）の階層を維持する。
2. **対象ファイル**: 現状は **JSON のみ**で移行する。将来 `public/data/rankings/` に CSV を置く場合は、同じルールで R2 にも同階層で追加すればよい（本 Phase の作業範囲外）。
3. **R2 上のキー（オブジェクトパス）**: **「パスは真実」** — ローカルのディレクトリ構造をそのまま R2 に写す。正規化・例外処理は行わない。プロキシは `getExternalRankingsUrl(relativePath)` で `{RANKINGS_BASE_URL}/data/rankings/{relativePath}` にアクセスするため、R2 には **`data/rankings/{年}/{リーグまたはシーズン}/{ファイル名}`** というキーで配置する（`public/` は含めない）。  
   - 例: ローカル `public/data/rankings/2025/PL/OPS.json` → R2 キー `data/rankings/2025/PL/OPS.json`
4. **非対象**: Next.js のページ・API・lib・計算ロジック・`output/backups` 等の生成物・バックアップは移行しない。デプロイ除外（`.vercelignore` の `public/data/rankings/`）は既存のままとする。
5. 上記を本計画書の「確定方針」および本 Phase の成果として Phase 2 以降で参照する。

**想定ネガティブポイントと解決策**

| ネガティブポイント | 解決策 |
|--------------------|--------|
| 年度・リーグによってディレクトリ階層が違う（例: 1936 は PRE_spring / PRE_fall、1950 は CL / PL） | **「パスは真実」**。ローカルにある構造をそのまま R2 に写す。正規化は不要（既存参照パスがその構造前提のため、変えると壊れる）。 |
| 「全指標」の定義がビルドスクリプトとずれる | 移行対象は「既に存在するファイルのみ」とする。不足分はビルド側で生成してから再度 Phase 3 でアップロードする。 |
| CSV を将来追加する場合の扱い | 確定方針のとおり、将来 CSV を置く場合は同じルールで R2 に同階層で追加する。現時点では JSON のみで進める。 |

## Phase 2: アップロード一覧の作成とサイズ・件数見積もり

**内容**

- `public/data/rankings/` 配下の全ファイルを一覧化し、総ファイル数と総サイズ（および必要なら年度別・リーグ別の内訳）を把握する。
- R2 の制限（オブジェクト数・ストレージ容量）およびアップロード手段（Wrangler / ダッシュボード / スクリプト）の制約に収まるか確認する。

**実施手順**

1. **移行前に必ず実施**: ローカルで `public/data/rankings` の**総サイズ**と**ファイル数**を一度だけ集計する（Cursor 等でコマンド実行で OK）。目安は約 879MB 級・R2 無料枠（10GB 想定）に収まる見込みだが、実測で確認する。
2. スクリプトまたは手動で `public/data/rankings/` を再帰的に走査し、ファイル一覧（相対パス）を出力する。相対パスは `data/rankings/` から始まる形にし、R2 キーと一致させる。
3. 総ファイル数と総バイト数を集計する。必要に応じて年度別・リーグ別の件数も出す。
4. Cloudflare R2 の無料枠・制限（オブジェクト数、ストレージ容量）と照らし、問題なければ Phase 3 のアップロード手段を選定する。認証は**手元 1 回のローカル手動アップロード**を想定（Wrangler CLI または R2 API 利用。トークンは環境変数 or パスワードマネージャで管理、リポジトリには入れない）。

**想定ネガティブポイントと解決策**

| ネガティブポイント | 解決策 |
|--------------------|--------|
| ファイル数が多く、一覧の生成やアップロードに時間がかかる | 一覧は一度だけ生成し、アップロードはバッチまたは並列数に上限を設けたスクリプトで実行する。途中失敗時は「未アップロード一覧」を出して再実行できるようにする。 |
| 総サイズが R2 の制限を超える | 年度単位で段階的にアップロードするか、R2 のプランを見直す。まずは見積もりで制限内であることを確認してから Phase 3 に進む。 |
| 日本語ファイル名（例: 打率.json）のエンコーディング | R2 は UTF-8 のオブジェクトキーをサポートする。スクリプトでは UTF-8 でキーを指定し、既存の URL エンコードと整合するか Phase 3 でサンプル確認する。 |

## Phase 3: R2 への一括アップロード（同一ディレクトリ構造）

**内容**

- Phase 2 で得た一覧に基づき、全ファイルを R2 にアップロードする。
- R2 上のキーは **`data/rankings/{年}/{リーグまたはシーズン}/{ファイル名}`** とし、ローカルの `public/data/rankings/` と同一の相対構造を維持する。

**実施手順**

1. アップロード手段を選定する（例: Wrangler CLI の `r2 object put` をループで実行するスクリプト、または Node/Python から R2 API を呼ぶスクリプト）。**認証**: 手元で 1 回実行。トークンはローカル環境変数 or パスワードマネージャで管理し、**リポジトリには絶対に入れない**。
2. ローカルパス `public/data/rankings/{年}/{リーグ}/{ファイル}` を、R2 キー `data/rankings/{年}/{リーグ}/{ファイル}` にマッピングしてアップロードする（**同一構造のまま**）。
3. アップロード後、サンプルパス（例: 2025/PL/OPS.json、1950/CL/打率.json）で R2 の公開 URL に直接アクセスし、期待する JSON が返ることを確認する。
4. 失敗したファイルがあれば一覧を残し、再実行または手動で補完する。

**再アップロード方針（確定）**: 当面は**全置換**推奨。運用が固まったら差分（sync）への移行を検討。削除されたファイルは **R2 からも削除**する（古い指標の残骸を残さない）。**R2 バックアップ**: 別バケット（例: rankings-backup）への定期コピーを推奨。まずは手動でよい。安定後に Workers/CI で自動化を検討。

**想定ネガティブポイントと解決策**

| ネガティブポイント | 解決策 |
|--------------------|--------|
| アップロード途中でネットワークや API 制限で失敗する | リトライとスキップ可能なスクリプトにし、未アップロード分だけ再実行できるようにする。 |
| 既存の R2 オブジェクト（例: test.json）を上書きしたくない | プレフィックスを分ける（例: ランキングは `data/rankings/` のみ）。test.json はバケット直下のため衝突しない。 |
| 日本語ファイル名が R2 または CDN で文字化けする | ブラウザとプロキシで実際にリクエストして確認する。問題があればキーを URL エンコードするか、ファイル名を ASCII に正規化する方針を Phase 1 に戻して検討する。 |

## Phase 4: 全年度外部参照の有効化（環境変数・プロキシ）

**内容**

- Vercel 上の本番環境で `RANKINGS_BASE_URL` を R2 の公開ベース URL に設定する（末尾スラッシュなし）。
- **全年度**を外部参照対象とするため、`RANKINGS_EXTERNALIZE_SCOPE` を**未設定または空**にする（scope 空 ＝ 全年度外部参照。本番の最終形）。
- **前提**: Phase 3 が完了し、**R2 側に全年度のデータが揃っている状態**で本 Phase を実行する。
- **サイトは既に Vercel でデプロイ済み**のため、既存プロジェクトに環境変数を設定し、Redeploy する。

**実施手順**

1. Vercel の対象 **Project** が既に存在することを確認する（デプロイ済みのため、そのまま Phase 4 の設定を行う）。
2. 対象 **Project** → **Settings** → **Environment Variables** で、**Production** に以下を設定する。  
   - `RANKINGS_BASE_URL` = `https://pub-41ff9f32fcf748529b7036f73f9e04e5.r2.dev`（または現在の R2 ベース URL）。末尾に `/` は付けない。
3. **全年度を R2 参照にする**ため、`RANKINGS_EXTERNALIZE_SCOPE` を**削除**するか、**空**にする。scope が空または未設定の場合、プロキシはすべての `/data/rankings/*` を R2 に転送する。
4. 保存後、**Redeploy** する。
5. デプロイ後、Phase 5 で代表年のみ確認して完了とする。

**想定ネガティブポイントと解決策**

| ネガティブポイント | 解決策 |
|--------------------|--------|
| scope を外したことで、R2 にまだないパスへのアクセスが 404 になる | Phase 3 で全ファイルのアップロード完了を確認してから Phase 4 を実行する。不足があれば Phase 3 で追加アップロードし、必要に応じて Phase 4 を再実行（Redeploy）する。 |
| 本番で RANKINGS_BASE_URL を誤って消してしまう | 手順書に「本番のランキングは R2 依存のため、この変数を削除しないこと」と明記する。削除した場合はすぐに再設定して Redeploy する。 |
| キャッシュで古いレスポンスが返る | プロキシの Cache-Control は既存のまま。必要なら一時的に revalidate を短くするか、デプロイ後にハードリロードで確認する。 |

## Phase 5: 動作確認とロールバック手順の整備

**内容**

- 複数年度・複数 season（CL/PL/PRE_spring/PRE_fall 等）について、サイトのランキング表示が R2 経由で正しく行われることを確認する。
- 問題発生時に、本番を戻す手順を文書化し、実行可能な状態にしておく。

**文書整備（実施済み）**

- **動作確認チェックリスト**と**ロールバック手順**の詳細は **`docs/phase5_verification_and_rollback.md`** にまとめた。  
  同ドキュメントに「代表年度・season の確認項目」「ネットワークで 200 確認」「案 A/B/C のロールバック手順」を記載している。

**要・人手実施（人間にしかできない部分）**

1. **動作確認**: Phase 4 実施後、**本番URL**（またはプレビューURL）を開き、`docs/phase5_verification_and_rollback.md` のチェックリストに従って**目視・手動**で確認する。  
   - 例: `/ranking/1936/PRE_spring`、`/ranking/1950/CL`、`/ranking/2025/PL` で表が表示されるか  
   - 開発者ツールの Network で `/data/rankings/...` が **200** で返っているか
2. **ロールバック手順の実行**: 問題発生時、同ドキュメントの「ロールバック手順」に従って、Vercel の環境変数変更や Redeploy を**人間が判断・実行**する。
3. **RANKINGS_BASE_URL の保護**: 本番のランキングは R2 依存のため、この変数を誤って削除しないよう注意する（削除時は即時再設定と Redeploy）。

**想定ネガティブポイントと解決策**

| ネガティブポイント | 解決策 |
|--------------------|--------|
| 一部年度・指標だけ 404 になる | Phase 3 のアップロード一覧と照合し、欠けているパスを R2 に追加する。 |
| ロールバック時にローカルにデータがない | 確定方針のとおり、**ローカルにデータが残っている前提**でロールバックする。日頃から手元の `public/data/rankings/` を削除しない、または別バックアップ（zip 等）を取っておく。 |
| パフォーマンス・r2.dev のレート制限 | r2.dev はレート制限があり本番推奨ではない。当面は **Vercel プロキシ＋Cache-Control** で吸収。最終的にはカスタムドメインを R2 に紐づける検討を推奨。 |

## 実行順序（サマリ）

1. **Phase 1**: 対象データと R2 パス仕様を確定し、文書化する。
2. **Phase 2**: ファイル一覧・件数・サイズを見積もり、アップロード手段を選定する。
3. **Phase 3**: 全ファイルを R2 に同一構造でアップロードし、サンプルで取得確認する。（※ 公開前でも実施可能）
4. **Phase 4**: 既存の Vercel 本番環境に `RANKINGS_BASE_URL` を設定し、`RANKINGS_EXTERNALIZE_SCOPE` を未設定または空にして Redeploy する。ランキングデータは Vercel に含めず R2 参照のまま。
5. **Phase 5**: 本番URLで代表年のみ表示確認し、ロールバック手順を整備する。

## 参照（旧・計画書作成時の疑問 — 確定済み）

以下は計画書作成時に挙がった疑問であり、本文の「確定方針」および各 Phase に反映済みである。
