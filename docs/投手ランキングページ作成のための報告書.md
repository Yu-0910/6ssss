# 投手ランキングページ作成のための報告書

## 目的

本資料は、**投手（投手成績）ランキングページ**を新規実装する担当者向けのガイドです。  
野手（打撃）ランキングで確立したデータフロー・UI・注意点を踏まえ、投手版で変更・追加すべき点を整理しています。**Phase 分け**と、野手版で発生した問題を振り返った**予想される懸念点・解決策**を記載しています。

**前提として読む資料**:  
`_data/reports/野手ランキングページ実装プロセス報告書.md`  
（スクレイピング〜UIまでの全フェーズと、発生しがちな問題31件が記載されています）

### 投手指標一覧（Record_pitching.csv）

サイトで扱う投手指標を `Record_pitching.csv` で定義する。**順番はこの通りに固定**し、UI の列順・ビルド・指標計算の基準とする。

| 順 | 指標 | 説明 |
|----|------|------|
| 1 | ERA | 防御率 |
| 2 | K-BB% | （計算: (SO−BB)/BF×100） |
| 3 | WHIP | （計算: (BB+H)/IP） |
| 4 | W | 勝利 |
| 5 | L | 敗戦 |
| 6 | G | 登板試合数 |
| 7 | IP | 投球回数 |
| 8 | SV | セーブ |
| 9 | BF | 対戦打者数 |
| 10 | H | 被安打 |
| 11 | HR | 被本塁打 |
| 12 | BB | 与四球 |
| 13 | IBB | 故意四球 |
| 14 | HBP | 与死球 |
| 15 | SO | 奪三振 |
| 16 | ER | 自責点 |
| 17 | R | 失点 |
| 18 | HOLD | ホールド |
| 19 | HP | ホールドポイント |
| 20 | CG | 完投 |
| 21 | SHO | 完封 |
| 22 | WPCT | 勝率 |
| 23 | K% | （計算: SO/BF×100） |
| 24 | BB% | （計算: BB/BF×100） |
| 25 | WP | 暴投 |

**計算が必要な指標**: ERA, K-BB%, WHIP, K%, BB% は生データ（ER, IP, BF, SO, BB, H 等）から算出する。Phase 2 の指標計算および Phase 4 の adapter で計算値を優先する。

### 前段階: スクレイピング準備

Phase 1 のスクレイピングを実施する前に、以下を行う。

- **URL の確認**: NPB公式の投手成績ページのURL構造を確認する（年度で変わる場合は `pit_p.html` / `pit_c.html` 等の分岐を把握）。
- **スクリプトの作成**: 投手用スクレイピングスクリプト（例: `scrape_npb_pitching_stats.py`）を作成する。野手版 `scrape_npb_batting_stats.py` をベースに、投手用の列マッピング・チーム略称マッピングを用意する。
- **列名マッピング**: 防御率／ERA、奪三振／SO／K、与四球／BB、投球回／IP、勝利・敗北・セーブ等の両表記に対応するマッピングを定義する。
- **出力先の決定**: 生CSVの保存先（例: `_data/master_csv__import_1950_2024/` または投手用サブディレクトリ）を決める。

前段階が完了したら、**Phase 1-a（テスト）** に進む。

#### 前段階 実施結果（実施済み）

| 項目 | 結果 |
|------|------|
| **URL** | 2024年以降: `https://npb.jp/bis/{year}/stats/pit_{p|c}.html`（PL=p, CL=c）。2023年以前: `https://npb.jp/bis/stats/{year}/{pl|cl}/pitching.html`（要検証）。 |
| **スクリプト** | `scripts/scrape_npb_pitching_stats.py` を作成。野手版をベースに投手用列マッピング（防御率/ERA、登板/G、勝利/W、敗北/L、セーブ/SV、投球回/IP、打者/BF、安打/H、本塁打/HR、四球/BB、死球/HBP、三振/SO、自責点/ER 等）とチーム略称マッピング（巨・神・ソ・西・オ・日・ロ・楽 等）を実装。 |
| **球団別ページ（idp1_*）の列マップ** | **実HTMLは先頭に空列あり**。選手名は index **1**。列14は常に投球回小数/空のため、H=15, HR=16, BB=17, SO=20, ERA=25 で固定（シフト不要）。`FIXED_COL_MAP_TEAM` は name:1, G:2, … H:15, BB:17, SO:20, … ERA:25。詳細は「課題4」「課題5」を参照。 |
| **出力先** | `_data/master_csv__import_1950_2024/pitching_{year}_{league}_from_master.csv`。 |
| **実行例** | `python scripts/scrape_npb_pitching_stats.py --year 2024 --league PL`（要: `pip install requests beautifulsoup4 lxml`）。 |

### Phase 1: スクレイピング

**実施順序**

1. **Phase 1-a テスト（2024年のみ）**  
   上記のスクリプトで **2024年・CL と 2024年・PL** の2ファイルを取得する。テストとして実施するが、**出力されたCSVは完成品としてそのまま本番に用いる**（破棄しない）。
2. **Phase 1-b 本格（2023年〜1950年）**  
   テストで問題がなければ、**2023年から1950年**まで同様に年度・リーグ別にスクレイピングを実行し、`pitching_YYYY_PL_from_master.csv` / `pitching_YYYY_CL_from_master.csv` を順次出力する。**最終的に全選手の成績を取得する**。取得方式は年度で分ける。**2005年〜2023年**は球団別 idp1_* から全投手を取得（実施済み）。**2004年〜1950年**は **2004年形式**（[「全ての選手から探す」](https://npb.jp/bis/players/all/index.html) 経由で年度・球団別選手一覧 → 各選手ページの投手成績表から該当年度行を取得）を**全年度に採用**する。これにより1950年までの全年度で全投手の成績を揃える。

**処理内容**

- NPB公式サイトの投手成績ページから HTML を取得し、テーブルをパースして生CSVを出力する。
- **取得対象**: その年度・リーグの**全投手**の成績を1ファイルにまとめる。規定到達・規定未到達の区別なく**全選手**を取得する。
- **取得方針（全年度）**: **最終的に全選手の成績を取得する**。**2005年〜2023年**は球団別 `idp1_{team_id}.html` から全投手を取得（実施済み）。**2004年〜1950年**は **2004年形式を全年度に採用**： [「全ての選手から探す」](https://npb.jp/bis/players/all/index.html) の年度別球団一覧 → 各球団の在籍選手一覧 → 各選手の `bis/players/{id}.html` の投手成績表から該当年度の行をパースし、CL/PL でまとめて出力する。同一の出力形式（`pitching_YYYY_LL_from_master.csv`）とする。
- 想定URL: 野手が `bat_{league_code}.html` の場合、投手は `pit_{league_code}.html` 等（要確認）。

**予想される懸念点**

| # | 懸念点 | 野手版での対応（参照） |
|---|--------|------------------------|
| 1 | **URL構造の変更**（年度でNPB公式のURLが変わる） | 野手では 2025 前後で URL が変更。投手も同様に年度判定で切り替えが必要（問題1） |
| 2 | **ヘッダー行の特定が困難**（テーブル内のどの行がヘッダーか判別しづらい） | 「選手」「順位」または「試合」「防御率」等が含まれる行をヘッダーと判定（問題2） |
| 3 | **列名の表記揺れ**（NPBは「奪三振」、CSVは「SO」など） | 防御率／ERA、奪三振／SO／K、与四球／BB 等の両表記に対応するマッピングを用意（問題3） |
| 4 | **player_id の取得失敗**（選手リンクからIDを取れない） | 正規表現で `/bis/players/数字` を検索し、フォールバック処理を実装（問題4） |
| 5 | **エンコーディング**（Windows で日本語が文字化け） | 標準出力を UTF-8 に設定（問題5） |
| 6 | **レート制限・タイムアウト**（NPB サーバーに負荷をかけすぎる） | 指数バックオフでリトライ、タイムアウト設定（問題6） |

**解決策（実施例）**

- 年度・リーグに応じた URL を設定ファイルまたは条件分岐で切り替える。
- ヘッダー判定は「選手」＋（「順位」or「試合」or「防御率」など）を含む行とする。
- 列名は日本語・英語・略称のマッピング表を用意し、投手用に「防御率」「ERA」「奪三振」「SO」「K」等を定義する。
- player_id 抽出は野手と同じ正規表現パターンを流用し、投手ページのリンク構造に合わせて調整する。
- スクリプト先頭で Windows 時は `sys.stdout` を UTF-8 にラップする。
- リクエスト間に sleep を入れ、タイムアウト・リトライを設定する。

#### Phase 1 補足: 野手版スクレイピングで実際に発生した課題と解決策

野手版（`scripts/scrape_npb_batting_stats.py` および `_data/reports/野手ランキングページ実装プロセス報告書.md`）では、Phase 1 のスクレイピングで**かなり苦労した**。以下はそのときに発生した課題と解決策を振り返り、投手版で同様のつまずきを避けるためにまとめたものである。

| 課題 | 症状・経緯 | 解決策 |
|------|------------|--------|
| **URL 構造の変更** | 2025 年以降、NPB 公式の成績ページの URL が変更された。旧形式 `.../stats/{year}/{league}/batting.html` では 2025 にアクセスできず 404 や別ページになる。 | 年度で分岐する。`year >= 2025` のときは新URL（例: `.../bis/{year}/stats/bat_{p|c}.html`）を使い、それ以前は旧URL。投手では `bat_` → `pit_` に読み替えて同様に年度判定を入れる。 |
| **ヘッダー行の特定** | ページに複数の `<table>` があり、さらに 1 つのテーブル内に「見出し」「サブヘッダー」「データ」が混在。どの行をヘッダーにすべきか判別できず、列マッピングがずれる。 | 投手では「選手」＋「防御率」または「試合」「投球回」等を含む行をヘッダーとする。**球団別 idp1_* の表**をパースするため、該当するヘッダー行を持つテーブルを特定する。 |
| **列名の表記揺れ** | NPB サイトでは「盗塁刺」、CSV や Record では「盗塁死」と表記が異なる。他にも日本語・英語・略称（打率／AVG、盗塁／SB 等）が混在する。 | ヘッダー文字列に対して、**複数の候補**（「盗塁死」「盗塁刺」「CS」等）で `in` または一致判定し、同じ論理列にマッピングする。投手では「防御率」「ERA」「奪三振」「SO」「K」「与四球」「BB」等を同様に複数候補で扱う。 |
| **player_id が取れない** | 2025 年など一部年度では、選手名のセルに `<a href="/bis/players/数字">` のリンクが**ない**ことがある。セル単位では ID を取得できない。 | (1) まず行内の `<a href="/bis/players/数字">` を探す。(2) なければ HTML 全文を正規表現（`/bis/players/(\d+)`）で走査し、「表示テキスト（選手名）」と player_id のマッピングを事前に作成。(3) データ行では選手名（括弧除去・正規化）でそのマッピングを引き、さらに**部分一致**（姓のみ一致等）でもフォールバックする。投手ページでも同じ方針で、リンクの有無に依存しない設計にする。 |
| **選手名の括弧とチーム** | 選手名列が「牧原　大成(ソ)」のように**括弧内にチーム略称**を含む形式。チーム列が別にある場合もあり、どちらを正とするかで混乱する。 | 括弧パターン `\(([^)]+)\)` で略称（巨・神・デ・ソ等）を取得し、略称→正式チーム名のマッピングで変換。選手名は括弧を除去して保存。チーム列が別にあればそれも参照し、空の場合のみ括弧から補完する。投手でも同じ形式なら同様のパースとマッピングを用意する。 |
| **複数テーブル** | 1 ページに成績テーブルが複数ある場合。 | **全選手は球団別 idp1_*（または全投手一覧ページ）から取得する**ため、メインページの「規定以上・セーブ上位・ホールド上位」の各表を個別に取得する必要はない。球団別ページの表をパースし、重複を避けて1つのCSVにまとめる。 |
| **エンコーディング** | Windows で標準出力に日本語を出すと cp932 で文字化けしたり、スクリプト実行結果のログが読めない。 | スクリプト先頭で `sys.platform == 'win32'` のとき、`sys.stdout` / `sys.stderr` を `io.TextIOWrapper(..., encoding='utf-8', errors='replace')` でラップする。取得 HTML は `response.encoding = response.apparent_encoding or 'utf-8'` でデコードする。 |
| **レート制限・タイムアウト** | 連続アクセスで NPB サーバーに負荷がかかり、ブロックされたり接続が切れる。 | `requests.get(..., timeout=30)` を指定。リトライ時は `time.sleep(2 ** attempt)` で指数バックオフ。User-Agent を一般的なブラウザに設定。投手用でも同じく 1 リクエストごとに間隔を空ける設計にする。 |
| **既存 CSV 更新時の重複** | 既存マスターに「新規選手のみ追加」する際、player_id が空の選手がいる。player_id だけでは重複判定できない。 | 重複判定を (1) player_id がある場合は「既存の player_id セットに含まれるか」、(2) player_id が空の場合は「(選手名, チーム名) のタプルが既存セットに含まれるか」の両方で行う。新規のみ追加し、既存行は上書きしない。投手でも同じポリシーにする。 |
| **球団別ページ（idp1_*）の列ずれ** | 球団別ページの表で選手数が明らかに少ない（約18行/球団がスキップされる）。 | **実HTMLは先頭に空列**があり選手名は index 1。列マップを「name: 1, G: 2, … ERA: 24」にし、name: 0 で読まない。左投手行は 0=*・1=名前のため列シフトは不要。スクリプトの `FIXED_COL_MAP_TEAM` を実HTMLに合わせて定義する。次回以降の実装・NPB の HTML 変更時は「課題4: 球団別ページの列ずれ」を参照。 |

**参照**: 実装の具体例は `scripts/scrape_npb_batting_stats.py` および野手報告書「フェーズ1: スクレイピング」「発生しがちな問題点（問題1〜6）」を参照のこと。

### Phase 2: 指標計算

**処理内容**

- 投手用 `Record.csv`（または `Record_pitching.csv`）の指標順に従い、各年度・リーグの生CSVを読み、ERA・WHIP・FIP・K/9・BB/9 等の派生指標を計算して計算済みCSVを出力する。

**予想される懸念点**

| # | 懸念点 | 野手版での対応（参照） |
|---|--------|------------------------|
| 7 | **Record.csv の読み込み失敗**（エンコーディング不一致） | 複数エンコーディング（utf-8-sig, utf-8, shift_jis, cp932）を順に試す（問題7） |
| 8 | **列名の正規化が不完全**（BOM・全角スペース・前後空白） | 正規化関数で BOM 除去・trim・全角スペース統一（問題8） |
| 9 | **計算値が NaN になる**（IP=0 で ERA 計算など） | 分母が 0 の場合は計算せず None／null。事前に分母チェック（問題9） |
| 10 | **依存指標の計算順序**（FIP が K/9, BB/9 等に依存する場合） | 依存する指標を先に計算し、順序を固定する（問題10） |
| 11 | **CSV の値が間違っている**（新年度データの不具合） | 計算値を優先し、CSV 値は検証してから使う（Phase 4 の adapter で対応）（問題11） |
| 12 | **ファイル名パースの失敗**（`pitching_2025_PL_from_master.csv` から年度・リーグを取れない） | 正規表現で `pitching_(\d{4})_(PL|CL)_from_master\.csv` をパース（問題12） |

**解決策（実施例）**

- 投手用 Record は複数エンコーディングで開く。列名は正規化関数を通してから使う。
- ERA = (ER×9)/IP、WHIP = (BB+H)/IP 等は、IP > 0 の場合のみ計算。それ以外は null。
- FIP 等は必要に応じて K/9, BB/9, HR/9 を先に計算し、同じスクリプト内で順序を明示する。
- 出力CSVは野手と同様「計算済みを優先」する前提とし、アプリ側（adapter）でも計算値優先とする。

### Phase 3: CSV読み込み

**処理内容**

- `loadPitchingCsv(year, league)` を実装し、計算済み → master → ルートの順で `pitching_YYYY_PL/CL_from_master.csv` を探索。投手用 Record で指標定義を読み、CSV に存在する指標のみを返す。

**予想される懸念点**

| # | 懸念点 | 野手版での対応（参照） |
|---|--------|------------------------|
| 13 | **CSV ファイルが見つからない**（年度・リーグの組み合わせでファイルがない） | 複数パスを探索（calculated → master → ルート）（問題13） |
| 14 | **エンコーディング判定の失敗**（UTF-8 で読んで文字化け） | 複数エンコーディングを順に試す（問題14） |

**課題1: pit_* 規定以上の表の列ずれ（全容）**

| 項目 | 内容 |
|------|------|
| **症状** | モイネロ行で IP=空・BF=163・WPCT=163.0 など、本来は IP=163, BF=636, WPCT=.688 となるべきところがずれていた。武内行で IP=0.1（本来は 145.1）など。 |
| **根本原因** | (1) **ヘッダーとデータ行のセル数差**: ヘッダー26セルに対しデータ行は27セル（チーム列などで1列多い）。そのため「ヘッダー基準の論理列」と「データ行の物理インデックス」が 1 ずれていた。(2) **固定マップの誤り**: 打者・投球回・勝率・安打・本塁打の**実インデックス**が、想定していた 25/26/27/28 用マップと一致していなかった。実HTMLでは 勝率=13, 打者=14, 投球回=15, (16=空または.1), 安打=17, 本塁打=18。(3) **27セル行でシフトしていた**: 投球回の次が .1 のときに「IP 以降を +1」するシフトを 27 セル行にも適用していたため、安打が 17→18 となり本塁打の値を安打として読んでいた。 |

**解決方法（実装済み）**

1. **実HTMLに合わせた固定マップの定義**  
   デバッグで得た実インデックスに基づき、FIXED_COL_MAP_27 / FIXED_COL_MAP_28 を次のように定義した。  
   - 勝率 WPCT=**13**、打者 BF=**14**、投球回 IP=**15**、安打 H=**17**、本塁打 HR=**18**、四球 BB=19 以降も同様に実位置に合わせる。  
   - 完投 CG=11、完封 SHO=12。
2. **27セル行ではシフトしない**  
   IP の次が .1 のときの「IP より後の列を +1」する処理を、**len(cells) >= 28 のときのみ**行う。27セル行では .1 は 16 番目にあり安打は 17 番目のため、シフトすると安打を誤読する。
3. **投球回の 2 セル結合**  
   _cell_val で、IP 列の値と次のセルが `^\.\d+$` のときは結合して float 化（例: 145 + .1 → 145.1）。  
4. **25/26 列用マップ**  
   他年度や別表でセル数が 25/26 のときに備え、FIXED_COL_MAP_25 / FIXED_COL_MAP_26 を維持。

**結果**: 2024 PL を再取得後、モイネロは G=25, IP=163.0, BF=636, WPCT=0.688、武内は IP=145.1, BF=568, H=120 となり、列ずれは解消している。

3. **結論**
   - **2004年について、NPB公式サイト上で「全投手の成績一覧」を提供しているURLは存在しない。**
   - 取得可能なのは年度別ページの「個人投手成績」**上位約10名/リーグ**（テーブル9）のみ。
   - 全選手を達成するには、(a) **他ソース**（他サイト・アーカイブ・手動データ）の利用、(b) テーブル3の特殊構造から「個人投手成績」部分だけを切り出してパースする検討、(c) **「全ての選手から探す」の活用**（下記）、のいずれかが必要。一括実行は**2004年の全選手取得が達成されるまで進めない**。

**2004年全選手取得のための代替手段（「全ての選手から探す」の活用）**

NPB の [個人年度別成績「全ての選手から探す」](https://npb.jp/bis/players/all/index.html) では**年度・球団で検索**でき、2004年を含む全年度で利用可能である。ここを取得元にすれば**全選手の成績**を賄える可能性がある。

- **URL と構造**
  - インデックス: `https://npb.jp/bis/players/all/index.html`
  - ページ内に「年度別球団一覧」があり、年を選択するとその年の球団リンクが表示される。
  - 各球団のリンク: `https://npb.jp/bis/players/search/yearly/{year}/{team_code}/`  
    例: 2004年・読売ジャイアンツ → `bis/players/search/yearly/2004/1947001/`、西武ライオンズ → `bis/players/search/yearly/2004/1979001/` 等（球団コードは年度により異なる場合あり）。
  - 球団ページにはその年に在籍した**選手一覧**（選手名・選手詳細へのリンク）が掲載される。各選手の年度別成績ページから**投手成績**を取得できる。

- **取得フロー（案）**
  1. インデックスページを取得し、対象年度（例: 2004）のセクションから**球団別URL一覧**をパースする。
  2. 各球団URLを取得し、**その年の在籍選手一覧**（選手ID・名前・年度別成績ページへのリンク）を取得する。
  3. 各選手の**年度別成績ページ**（例: `bis/players/{player_id}/yearly/{year}.html` 等）を取得し、投手成績があればパースして行として追加する。
  4. 球団を **CL / PL** に振り分け（2004年の12球団対応表を用意）、`pitching_2004_PL_from_master.csv` / `pitching_2004_CL_from_master.csv` 形式で出力する。

- **計画上の位置づけ**
  - 上記フローをスクリプト化し、**2004年**で「全ての選手から探す」経由の全投手取得を実装済み。
  - **2004年形式を 2003年〜1950年までの全年度に採用する**。同一スクリプトを年度パラメータで 2003, 2002, …, 1950 と実行するか、一括実行オプションを追加する。取得件数・列・選手IDは既存の `pitching_YYYY_LL_from_master.csv` 仕様に合わせる。
  - 年度ごとに球団構成・球団コードが異なるため、インデックスページから対象年度の球団URL・球団名→リーグの対応を取得する処理を年度対応させる。

**2004年 完成品の評価**

「全ての選手から探す」経由の実装（`scripts/scrape_2004_pitching_via_all_players.py`）により、2004年は**全投手**の成績が取得された完成品として評価する。

| 項目 | 内容 |
|------|------|
| **取得件数** | PL **151件**、CL **150件**、計 **301件**。 |
| **比較** | 年度別ページのみの約20名/リーグから、**全投手レベル**に到達。2005年（PL 142・CL 146）と同程度の規模であり、12球団の在籍投手を網羅していると判断できる。 |
| **取得元** | [全ての選手から探す](https://npb.jp/bis/players/all/index.html) → 2004年球団別URL → 各選手の `bis/players/{id}.html` の投手成績表から2004年行をパース。 |

**2004年以前（2004〜1950）の取得方式（採用方針）**

**ホールド導入前行の H/HP 扱いと Phase 1 再スクレイピング計画**

NPB 選手ページの投手表は、**ホールド導入前**（おおよそ1980年代〜1990年代以前）の年度では、ヘッダーに「H」「HP」があるにもかかわらず**データ行には H・HP 列が存在しない**。このため、ヘッダー基準の列マッピングで読むと、完投・完封以降の列が2列ずれ、**失点が防御率(ERA)に、自責点が失点に**読まれてしまう（ERA 異常値の原因）。詳細は `docs/H欠損とERA異常値_原因究明レポート.md` を参照。

| 方針 | 内容 |
|------|------|
| **H/HP を「0」とみなす** | ホールド導入前の行では H（ホールド）・HP（ホールドポイント）は本来 0。データ行に列が無い場合、**こちら側で 0 として扱う**。 |
| **派生指標への影響** | **影響なし**。`Record_pitching.csv` の計算指標（ERA, WHIP, K%, BB%, K-BB%, WPCT）はいずれも HOLD・HP を使用しない。これらは ER, IP, BF, SO, BB, H（被安打）等から算出する。 |
| **実装** | `scrape_2004_pitching_via_all_players.py` で、データ行のセル数がヘッダーより少ない場合（ホールド導入前形式）を検出し、(1) HOLD=0, HP=0 を設定、(2) 完投〜防御率の列に **オフセット -2** を適用して正しい値（失点・自責点・防御率）を読む。 |

## 5. 野手で起きた問題から投手で特に気をつけること

1. **CSVの値は信用しない（特に新年度データ）**  
   計算可能な指標（ERA, WHIP, FIP, K/9 等）は必ずアプリ側で再計算し、計算値を優先する。

2. **列名の表記揺れ**  
   日本語（防御率・奪三振等）と英語（ERA, SO 等）の両方、および NPB サイトと CSV の表記差に対応するマッピングを用意する。

3. **規定の基準の違い**  
   野手は「規定打席」、投手は「規定投球回」または「試合数」。年度・リーグごとの規定値テーブルを用意する。

4. **adapter と leaders のロジックを一致させる**  
   ランキング表とトップページのリーダーで同じ計算・同じ列を使うこと。

5. **Record（指標順）を唯一の正とする**  
   投手用の指標リストも「Record の順」で並べ、UI の列順はそれに従う。

6. **球団名が時代によって変わる**  
   合併・名称変更のため、1950 年代等のデータでは「大映」「毎日」など現在と異なる球団名が使われる。規定判定・表示・重複判定で一貫させるため、年度・時代別の「旧表記 → 現在の表記」マッピングを用意する。野手版では `lib/ranking/qualifyingPA.ts` および `scripts/add_qualifying_pa_flags.py` でマッピングを定義している。投手版でも同じマッピングを流用する。

- **Phase 2 以降**: 未着手。データが揃い次第、野手版のパイプラインに投手用を載せる形で進められる。
- **推奨する次のアクション**:  
  1. **2003→2000** を `scrape_2004_pitching_via_all_players.py --year` または **`run_phase1b_2000_2003_via_all_players.bat`** で実行し、2004年形式の CSV で上書きする。日本語を含むプロジェクトパスの場合はバッチをエクスプローラーから実行すること（`docs/日本語パス問題の原因と対処.md` 参照）。  
  2. 必要に応じて **1999→1950** を同スクリプトで順次実行する（古い年度では球団マップの動作確認を推奨）。  
  3. Phase 1 が全年度揃った時点で、Phase 2（指標計算）に進む。

以上を「計画のここまでの評価」とする。

## 8. まとめ

- 野手ランキングは「CSV → loaders → adapter → 規定 → ViewModel → 同一UI」の一連の流れで動いている。
- 投手版は **Phase 1〜7** に沿って、**同じパイプライン**に「投手用CSV・投手用指標・規定投球回・投手用計算」を載せる形で実装できる。
- 各 Phase の懸念点・解決策は野手報告書の「問題1」〜「問題31」を投手用に読み替えたものである。詳細は `_data/reports/野手ランキングページ実装プロセス報告書.md` を参照し、投手用に適用する。
