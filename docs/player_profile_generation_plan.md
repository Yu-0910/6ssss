# プレイヤーページ 選手プロフィール欄 生成計画書

## 1. 概要

プレイヤーページに表示する選手プロフィール欄を、NPB公式を中心にデータ取得しつつ、取得不可能な項目は外部データ・計算で補完して生成する。**段階的（Phase 1〜6）に実装**する。

| 項目 | 表示例 |
|------|--------|
| 生年月日 | 1994年11月9日（30歳） |
| 出身地 | 兵庫県津名郡東浦町 |
| プロ入り | 2018年 ドラフト2位（全体14位） |
| 経歴 | 社高等学校 → 関西学院大学 → 大阪ガス → 阪神 (2019 - ) |
| 生涯年俸 | 10億5000万円 |
| チーム成績 | 日本一：5回、リーグ優勝：7回 |
| FA取得（推定） | 2027年 |

### 1.2 フェーズ一覧

| Phase | 内容 | 成果物 |
|-------|------|--------|
| **Phase 1** | NPB公式スクレイピング（生年月日・出身地・プロ入り・経歴） | プロフィール基盤データ |
| **Phase 2** | ドラフト全体順位（④） | プロ入り欄に「全体○位」付与 |
| **Phase 3** | チーム成績（②） | 日本一・リーグ優勝回数 |

## 3. Phase 1：NPB公式スクレイピング（プロフィール基盤）

### 3.1 範囲

NPB公式サイトの選手プロフィールページ（例: `https://npb.jp/bis/players/01705130.html`）から以下を取得する。

- **生年月日** … ページ記載の生年月日。表示時は「YYYY年M月D日（N歳）」形式（年齢は計算）。
- **出身地** … 出身地・出身校等の記載をそのまま利用。
- **プロ入り（ドラフト情報）** … ドラフト年・球団内順位（例：ドラフト2位）まで取得。「全体○位」は Phase 2 で付与。
- **経歴** … 高校→大学→社会人→プロの経歴テキスト。

### 3.2 成果物

- 選手プロフィール基盤データ（player_id, name, birth_date, birth_place, draft_year, draft_round_rank, career 等）。
- 既存の選手ID・名前との紐付け方法の確定。

### 3.3 想定される問題

| 問題 | 内容 | 対処案 |
|------|------|--------|
| **選手IDの取得方法** | NPB公式の選手ページURL・ID体系が年度やリニューアルで変わる可能性がある。 | URLパターン・ID一覧の取得元をドキュメント化し、変更時にスクリプト修正箇所を限定する。 |
| **ページ構造の変更** | NPB公式のHTML構造が変わるとスクレイパーが壊れる。 | セレクタを設定ファイル化し、定期的な動作確認・ログ監視を検討する。 |
| **経歴・出身地の表記ゆらぎ** | 「兵庫県津名郡東浦町（現：淡路市）」など表記が年度・ページで異なる。 | 正規化ルールを決める（例：括弧内はそのまま／省略しない）。欠損時は「-」表示など方針を決める。 |
| **アクセス頻度・規約** | 大量スクレイピングでブロック・規約違反のリスク。 | 間隔を空ける、robots.txt 確認、必要なら公式API・データ提供の有無を確認する。 |
| **ドラフト未加入選手** | 外国人選手・入団経路が特殊な選手は「プロ入り」が空になる。 | フィールドを nullable とし、表示時は「-」や「入団年度のみ」などルールを決める。 |

## 4. Phase 2：ドラフト全体順位（④）

### 4.1 結論（採用策）

- **各年のドラフト指名一覧（支配下）を、指名順に上から並べる**。
- **行番号＝全体順位**とする。
- 「(巡目−1)×12＋○」のような計算式は**使わない**（指名人数・指名終了が年ごとに異なるため）。

### 4.2 実装ポイント

- **支配下**と**育成**は別枠で管理する。
  - 支配下の全体順位: `overall_pick`
  - 育成の全体順位: `overall_pick_dev`（必要なら別表示）
- データは「年度・指名順・球団・選手」の一覧を持ち、その順序で overall_pick を付与する。
- Phase 1 の「プロ入り」に「ドラフト○位（全体○位）」を結合する。

### 4.3 表示例

- 「2018年 ドラフト2位（全体14位）」
- 育成の場合は「2018年 育成ドラフト○位（全体○位）」等、表示方針を別途定義。

### 4.4 データフロー

```
NPB公式または信頼できるソース: 当該年度の支配下ドラフト指名一覧（指名順）
  → 1行目=1位、2行目=2位、… として overall_pick を付与
  → 選手ID・名前と紐付け
  → プロフィール「プロ入り」欄に「ドラフト○位（全体○位）」を結合
```

### 4.5 想定される問題

| 問題 | 内容 | 対処案 |
|------|------|--------|
| **指名一覧のデータソース** | NPB公式に年度別「指名順」の一覧が揃っているか不明。 | 公式ドラフト結果ページの構造を確認し、ない場合はスポーツ紙等の一覧を補助的に利用する方針を検討。 |
| **選手名の表記差** | ドラフト一覧の名前とNPB選手ID・Phase 1 の名前で表記が違う（スペース・長音等）。 | 正規化（トリム・全角統一・長音統一）とフォールバック照合（読み・ローマ字）を検討。 |
| **重複指名・競合** | 同一選手が複数球団に指名された場合の扱い。 | 実際に契約した球団の「指名順」のみを採用する等、ルールを明文化する。 |
| **育成→支配下** | 育成で入団しのち支配下になった選手の「プロ入り」表示方針。 | 支配下転向年と入団年を分けて持つか、「最初の支配下契約」をプロ入りとするか等、表示ポリシーを決める。 |

## 5. Phase 3：チーム成績（②）

### 5.1 結論（採用策）

1. **NPB公式の年度別優勝球団データを全年度取得**する。
2. **選手の在籍年（年別所属球団）**をNPB公式の年度別成績・所属情報から取得する。
3. 在籍年と優勝年が**重なった年数だけ**日本一・リーグ優勝をカウントする。

### 5.2 実装ポイント

- 判定は**年単位**で行う（途中移籍年は簡略化してよい）。
- 出力は次の2つ: **日本一回数**、**リーグ優勝回数**。
- **データ源はNPB公式のみ**とし、確定情報として扱う。

### 5.3 データフロー

```
NPB公式: 年度別リーグ優勝球団・日本一球団
NPB公式: 選手の年度別所属球団（成績ページ等）
  → 年度ごとに「その年に在籍していた球団が優勝/日本一だったか」を判定
  → 該当年数をカウント
  → 日本一：○回、リーグ優勝：○回
```

### 5.4 想定される問題

| 問題 | 内容 | 対処案 |
|------|------|--------|
| **在籍年の定義** | 開幕時所属か、シーズン中在籍日数かで変わる。 | 「その年に1軍または2軍で在籍していた球団」等、ルールを1つに固定しドキュメント化する。 |
| **途中移籍** | シーズン途中でトレードされた年は両球団のどちらでカウントするか。 | 簡略化として「その年に在籍したいずれかの球団が優勝/日本一なら1回カウント」等、二重カウントしないルールを決める。 |
| **優勝データの取得** | NPB公式の年度別優勝・日本一の一覧がどこにあり、どう取得するか。 | 公式の成績・歴代優勝ページを特定し、スクレイピングまたは手動CSVの更新手順を決める。 |
| **合併・休止年度** | 過去の合併リーグ・休止年度がある場合の扱い。 | 対象年度を限定するか、優勝データがない年度はスキップする等、範囲を明示する。 |

## 6. Phase 4：生涯年俸（①）

### 6.1 結論（採用策）

- **外部データ**: baseball-data.com を使用する。
- 年別の推定年俸を収集し、合算して「生涯年俸（推定）」として掲載する。

### 6.2 実装ポイント

- 年俸はNPB公式データではないため、**必ず「推定」「出典（baseball-data）」を明記**する。
- データ形式: `salary_YYYY.csv` を年度ごとに用意し、選手ID・名前等で紐付け。
- 集計: 該当選手の全年度年俸を合算 → `career_total_salary_est` として保持。
- 表示例: 「生涯年俸（推定） 10億5000万円（出典: baseball-data）」

### 6.3 データフロー

```
salary_2019.csv, salary_2020.csv, ... 
  → 選手ごとに年度別年俸を取得 
  → 合算 
  → career_total_salary_est（数値または「○億○万円」文字列）
```

### 6.4 想定される問題

| 問題 | 内容 | 対処案 |
|------|------|--------|
| **baseball-data.com の提供形式** | API／CSV／スクレイピングのいずれか、利用規約・更新頻度が不明。 | 利用条件・データ形式を確認し、規約に沿った取得方法と更新スケジュールを決める。 |
| **選手の紐付け** | 外部サイトの選手ID・名前がNPB公式や自前IDと一致しない。 | Phase 1 の選手マスタと名前・年度で照合し、手動マッピング表を用意する等で対応。 |
| **年度欠損** | ある年度だけ年俸データがない選手が出る。 | 「データがある年度のみ合算」とし、表示時に「（該当年度のみ）」等の注釈を検討。 |
| **単位・通貨** | 万円／円の混在、過去の改定。 | 内部は円または万円に統一し、表示時に「億」「万」でフォーマットする。 |

## 7. Phase 5：FA取得年（③）

### 7.1 結論（採用策: B案＝推定）

- 本来は「1軍登録日数」で判定するが取得不可のため、**成績（登板数・投球回／出場試合・打席）から推定**する。
- 年度ごとに **0 / 0.5 / 1.0 年** で年換算し、累積してFA取得年を算出する。

### 7.2 投手（確定基準）

| 換算 | 条件 |
|------|------|
| **1.0年** | IP≥80 **OR** G≥40 **OR** (G≥25 **かつ** IP≥50) |
| **0.5年** | IP≥40 **OR** G≥20 **OR** (G≥15 **かつ** IP≥25) |

### 7.3 野手（確定基準）

| 換算 | 条件 |
|------|------|
| **1.0年** | 90試合出場 **OR** 340打席 |
| **0.5年** | 45試合出場 **OR** 170打席 |

### 7.4 2020年特例

- 試合数短縮のため、**全基準を「120/143」で比例縮小（切り上げ）**して適用する。
  - 例: 野手 1.0年 → 90×(120/143)≒76 試合 または 340×(120/143)≒286 打席（切り上げで運用）。

### 7.5 取得判定

- 年換算を**累積**し、次の年数に到達した**年度**を「FA取得年（推定）」とする。
  - **国内FA**: 7年 または 8年（入団経路等により異なる場合は別ロジックで切り替え）
  - **海外FA**: 9年

### 7.6 表示・注意

- **必ず「推定（成績ベース）」と明記**する。
- 表示例: 「FA取得（推定） 2027年（成績ベース推定）」

### 7.7 想定される問題

| 問題 | 内容 | 対処案 |
|------|------|--------|
| **投手・野手の判定** | 二刀流やポジション変更で年度によって投手・野手が変わる。 | その年度の登板数・打席数等で「主に投手か野手か」を判定するルールを決める（例：登板あり→投手基準）。 |
| **国内7年/8年の切り替え** | 高校卒9年・大卒8年・社会人7年等、経路で年数が違う。 | Phase 1 の経歴から入団経路を推定するか、経路マスタを用意して年数テーブルを引く。 |
| **成績データの出所** | 打席・試合数・投球回等をどこから取るか（既存CSV／NPB公式）。 | 既存の打撃・投手成績CSVと整合し、欠損年度は0年扱いにする等ルールを統一する。 |
| **2020年以外の短縮** | ストライキ等で他年度も試合数が少ない場合。 | 2020年のみ特例として実装し、他年度は通常基準。必要なら年度別係数を後から追加。 |
| **累積が境界付近** | 7.0年に達するのがシーズン途中の扱い。 | 年度単位で「そのシーズン終了時点で累積≥7」ならその年をFA取得年とする等、ルールを固定する。 |

## 8. Phase 6：プレイヤーページUI統合

### 8.1 範囲

- Phase 1〜5 で整備したプロフィール用マスタ（またはAPI）を読み込む。
- プレイヤーページのプロフィール欄に表示する。
- **推定値には「（推定）」、非公式データには「出典: ○○」を漏れなく表示する。**

### 8.2 成果物

- プロフィール欄の表示コンポーネント（項目ごとの表示・欠損時の「-」等）。
- 表示ルールのドキュメント（推定・出典の表記）。

### 8.3 想定される問題

| 問題 | 内容 | 対処案 |
|------|------|--------|
| **データ未整備の選手** | まだPhase 1〜5のデータがない選手がURLでアクセスされた場合。 | プロフィール欄は「準備中」または該当項目のみ「-」表示とし、ランキング等からのリンク先はデータがある選手に限定するか検討。 |
| **マスタとランキングの選手ID** | ランキングで使っている playerId とプロフィールマスタの player_id が一致しない。 | 両者を対応させるIDマッピング表または統一IDを導入する。 |
| **キャッシュ・更新** | プロフィールデータの更新頻度と、ビルド時／実行時の読み込み方。 | 静的JSONをビルドに含めるか、API化するか決め、更新手順とキャッシュ方針を書く。 |

## 9. データ構造・ファイル案

### 9.1 プロフィール用マスタ（例）

- `_data/player_profile/` 以下に配置する想定。
- Phase 1〜5 の結果をマージして1レコードにする。

| フィールド例 | 説明 | ソース（Phase） |
|--------------|------|------------------|
| player_id | 選手一意ID（NPB公式ID等） | NPB（1） |
| name | 氏名 | NPB（1） |
| birth_date | 生年月日 | NPB（1） |
| birth_place | 出身地 | NPB（1） |
| draft_year | ドラフト年 | NPB（1） |
| draft_round_rank | 球団内順位（例: 2） | NPB（1） |
| draft_overall_pick | 全体順位 | ④（2） |

| career_total_salary_est | 生涯年俸（推定）円 or 表示用文字列 | ①（4） |

### 9.2 参照データ

- **年俸**: `salary_YYYY.csv`（年度別、選手ID・年俸等）… Phase 4

- **ドラフト**: 年度別支配下指名一覧（指名順）、`overall_pick` / `overall_pick_dev` … Phase 2

## 10. 表示上の注意（共通）

- **推定値**には必ず「（推定）」を付ける。
- **非公式データ**（例: 生涯年俸）には「出典: baseball-data」等を明記する。
- **FA取得年**には「成績ベース推定」である旨を明記する。
- NPB公式のみに基づく項目（生年月日・出身地・プロ入り・経歴・チーム成績）は、出典を「NPB公式」として扱い、必要に応じて表記する。
